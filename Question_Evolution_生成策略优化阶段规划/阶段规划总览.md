# Question Evolution 生成策略优化阶段规划总览

## 总体目标

本次优化只改造 question evolution 的生成策略，将现有“高分样本 -> 统一 Prompt -> 进化题 -> 重新评分 -> 看平均分”升级为“样本诊断 -> 能力轴识别 -> 算子路由 -> 有状态进化 -> 复杂度校验 -> 重新评分 -> 轻量效果统计”。

本阶段不优化 `gen_rubric.py`、rubric prompt、score prompt、rubric item、权重、扣分项或 judge 可信度建设。`collect_answers.py`、`gen_rubric.py`、`scoring.py` 仍作为闭环执行步骤保留。

## 阶段划分

1. 阶段01：数据合约与基础结构
2. 阶段02：样本画像与候选分流
3. 阶段03：算子路由与多算子生成
4. 阶段04：复杂度校验与候选选择
5. 阶段05：效果统计状态更新与记忆库
6. 阶段06：流水线集成与验收回归

## 推荐实施顺序

按阶段01至阶段06顺序执行。阶段02和阶段03的 prompt 文案可以并行草拟，但代码接入必须等待阶段01字段合约稳定后进行。阶段06依赖前五个阶段的产物，不建议提前实施。

## Codex Goal 建议

每个阶段都适合作为一次独立 Codex Goal 执行。阶段03范围最大，如果需要降低风险，可拆成“router + 首批核心算子 O1/O2/O4/O8/O9”和“补齐 O3/O5/O6/O7”两个执行任务。其余阶段不建议再拆碎，避免按文件级微任务推进造成上下文浪费。

## 需要人工确认的事项

- `admitted_seed_samples.jsonl` 的实际路径是否替换当前 `run_loop.sh` 默认输入。
- profile、router、evolution 使用的模型、base URL 与 API key 来源是否统一。
- `lightweight_boundary_hit` 与 `hit_confidence` 的人工复核口径。

## 风险边界

- `expected_evaluation_focus` 只能作为问题生成元数据，不能传入 rubric 生成或评分 prompt。
- 轻量效果统计不读取 rubric item，不做 judge agreement，不做 rubric drift。
- operator memory bank 中低置信命中必须带 `needs_manual_review=true`，不能沉淀成强成功经验。
