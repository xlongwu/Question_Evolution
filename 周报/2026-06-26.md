# 存在问题
1. 当前PoliceQA 数据难度进化原始脚本对未进化样本仍会进行answer、rubric重新采集，影响最终结果（已修改脚本）；
2. 原始进化prompt存在问题：把题目推向了“超长、多模块、强格式、强硬限制”的方向；虽然写了“选择 1-3 种策略，不要全部堆砌”，但实际结果显示，模型经常把“增加约束 + 显式推理 + 反事实 + 综合复杂度 + 固定格式”同时叠加。优化的prompt在实验后发现依赖“边界判断”这一类题型、对已经能稳定处理边界题的样本，缺少进一步转向机制、压分来的慢等问题；
3. 初步实验发现Qwen模型存在的能力不足的地方：
   一道题里摆出几个都像缺口的地方时，Qwen 不能找准最影响结论的那个；
   题干只多加一条新信息时，Qwen 不总能把原来的判断顺序重新排对；
   经常能说出“还缺证据”，但说不准最少到底缺哪一条；
4. Qwen当前做的好的地方：
   题目就算越改越长，Qwen 通常也能把答案按照要求写出来；
   题目要求按步骤说、按层说时，Qwen 往往答得比较好；
   只要核心问题是“现在能不能直接下结论”，Qwen 往往能给出正确回答；
   基础边界判断类别不是弱项能力；

# 想法或疑惑
1. 当前挑选的子集样本不具备针对性，后续可能难以测出真实边界能力；
2. 需要进一步从进化方向、进化机制展开思考。例如直推链式到可回溯树形结构、进化的思路在其余样本间进行复用；
3. 进一步调研数据进化pipeline的发展；
4. 某种进化方向以及思路并不能在各种类型的样本上有效；

# 当前项目的问题思考
1. 进化方向、进化机制（链式，树状）、进化时在某个问题上得到的经验可以复用
适合这个项目的不是纯链式，也不是纯树状，而是“链式主流程 + 局部树状探索”。链式适合做主迭代，因为你最终要落成单条样本的稳定版本；树状适合在单个样本上并行尝试 2-4 个进化方向，再选最有效的一个进入下一轮。经验复用要显式化，不能只靠模型“记住”。建议给每个成功进化样本沉淀一条结构化经验：
  - 原题画像
  - 候选答案虚高主因
  - 使用的进化算子
  - 进化后主要压出的错误
  - 是否成功降分
  - 是否造成不可回答/复杂度膨胀
  这会形成一个“operator memory bank”，后续先检索相似样本再决定怎么进化。

2. 每个问题考察的维度（类型，概念，组合问题）以及最终评分的情况
这个项目现在最缺的是“题目画像 schema”。我建议每题在进入 pipeline 前先打上结构化维度，不只是已有 labels，还要增加：
  - 核心能力类型：概念识别 / 证据链补强 / 时空关联 / 边界判断 / 反事实 / 排他性认定 / 程序规范
  - 结论层级：事实识别 / 线索判断 / 高度怀疑 / 可写入结论
  - 题型结构：单概念 / 双概念比较 / 多条件组合 / 多阶段流程
  - 候选答案常见虚高模式：套话罗列 / 层级越推 / 题外补设 / 抓显眼点漏关键点
评分也不要只看总分率。至少要同时看：
  - score_rate 变化
  - 失分是否集中在 1 个核心错误
  - judge 是否稳定
  - evolved 后 rubric 是否更尖锐而不是更长
  - 是否出现“题更长了，但失分没有更可解释”

3. （原始数据）问题的选择进行筛选，那些容易区分的不必加入进化pipeline
不是所有高分题都值得进化，也不是所有原始题都该进 pipeline。建议做两层筛选：
  - 进入 pipeline 前先排除“显然定性题”“纯定义题”“答案高度模板化但并不需要深推理的题”
  - 进入 pipeline 后只保留“高分且高分原因可疑”的题，而不是所有高分题更具体地说，优先保留这几类原题：
  - 需要从视频线索过渡到行为性质判断
  - 需要区分“线索”“可疑”“可写结论”
  - 需要比较两个看起来都像关键缺口的证据点
  - 需要处理监控盲区、时间差、跨点位拼接、证据固定边界
不建议优先进化：
  - 一眼就能定性的题
  - 单纯背定义的题
  - 只要罗列流程就能高分、但业务价值不高的题

4. 问题长度不是一定要长或短，而是结合实际情况
长度不是目标，信息密度和主轴清晰度才是目标。建议把“长度控制”改成“复杂度预算控制”，而不是硬控长短。预算维度至少包括：
  - 主轴数
  - 新增事实数
  - 输出任务数
  - 候选项数
  - 反事实数
  - 硬约束数
实验文档里已经在提这个方向，但还需要代码级校验，不然多轮链式进化很容易越改越肿。

5. 调研进化pipeline，别人是如何做的，进化机制，进化方向的考量
和你项目最相关的几条路线是：
  - Self-Instruct：核心是“自举生成 + 去重过滤 + 再利用”。
  - Evol-Instruct / WizardLM：核心是“从种子题逐步改写成更复杂题”，非常接近你们当前的question evolution。
  - STaR：核心是“只保留真正成功的推理轨迹，再拿这些成功经验反哺下一轮”。
  - Tree of Thoughts：核心是“不要只走一条改写路径，允许多分支探索后再选”。
  - Reflexion / Promptbreeder：核心是“把失败反馈和 prompt 演化本身也纳入记忆与优化对象”。
对当前项目最有用的不是照搬，而是吸收 3个原则：
  - 进化前先分类，不同样本走不同算子
  - 进化后要过滤，不能把所有改写都当有效样本
  - 经验要沉淀成可检索的显式记忆，而不是散落在实验结果里

6. 某种进化方向以及思路并不能在各种类型的样本上有效，如何解决这些问题
不要追求“一个万能evolution prompt”。更合理的是做“算子路由”：
  - 对“结论层级混淆”样本，用“子判断定位 / 单步跳跃识别”
  - 对“多个缺口都像关键点”样本，用“候选缺口二选一 / 近似项分层”
  - 对“题外补设”样本，用“题干外补设识别”
  - 对“泛化套话”样本，用“具体化约束”
  - 对“边界条件敏感”样本，用“单变量反事实”，然后维护一张 sample_type x operator 的效果矩阵，统计哪类样本被哪类算子稳定压分，失败时就换算子，不要在同一路线上反复加料。

## 当前项目还需要重点思考的地方
- Judge 可信度
现在项目默认是 Qwen 自己答、Qwen 自己judge，这个风险很大，容易出现“风格自洽高分”。这是当前最需要警惕的系统偏差，见
scoring.py 和 readme.md。至少要做一部分双judge 校准。
- Rubric 漂移
当前已经意识到这个问题，并在实验分析里强调不要拿未进化样本来判断 evolution prompt。这是对的。但还要继续做“rubric 稳定性检查”，否则分数变化可能来自 rubric 变化，不是来自题变尖了。
- 成功标准
不要把“平均分下降”当项目成功。更好的标准是：
   - 失分是否可解释
   - 是否稳定集中在 1 个能力边界
   - 是否减少满分虚高样本
   - 是否没有显著增加不可回答样本
- 停止条件
一个样本不能无限进化。建议设 stop rule：
   - 已经出现不可回答风险
   - 题长/复杂度超预算
   - 连续两轮没有带来更清晰失分
   - 已经压出稳定的单一错误
- 数据资产沉淀
最终最有价值的不是“这一轮又生成了多少题”，而是沉淀出一批：
   - 高价值原题画像
   - 有效进化算子模板
   - 稳定边界样本
   - 无效复杂化反例
后者同样重要，因为它能告诉你哪些进化方向不要再浪费算力。

## 把项目做好还需要考量的点
- 先建“样本准入规则”，减少不值得进化的题。
- 再建“样本画像 + 算子路由”，不要用单一进化方向覆盖所有题。
- 把 prompt 里的复杂度预算变成代码里的硬校验。
- 给每轮进化结果做失败归因：有效压测、无效复杂化、不可回答、judge 不稳定。
- 做经验库，而不是只存 jsonl 成果。
- 保留一小批人工复核样本，专门看 judge 偏差和 rubric 漂移。
- 用“边界命中率”替代“平均降分”作为核心优化目标。


## 建议优化方向
1. 给原始样本补一层“进化前画像标签”。
2. 把进化方式拆成若干 operator，并做路由而不是单 prompt 通吃。
3. 在代码里加 validate_evolved_question()一类的硬检查。
4. 建一个 经验库/失败库，记录“什么样本适合什么算子”。

## 为什么排除了其他看起来也不错的题

### 1. 容易一眼收敛的题

### 2. 纯概念或定义色彩偏强的题

例如“某类人员的定义是什么”“某方法的概念是什么”这类题，即使可以扩展，也不符合当前 question evolution 的主攻方向。

### 3. 单纯偏程序或背诵的题

如果一道题主要靠列流程、列注意事项就能高分，但缺少明显的边界判断、证据层级或排除性推理空间，本轮也不优先纳入。

### 4. 题面过于依赖题外补充的题

如果某题要高质量回答必须补充大量题外专业背景，容易在后续进化时退化为“靠外部知识填坑”，这类题也不优先纳入。

1. 参考答案中的核心知识和推理链条仍然适用，或只需合理扩展即可回答新题；
2. 候选答案中的 superficial / 模板化 / 泛化 / 堆叠术语式的回答不再能轻易得高分；
3. 新题能迫使回答展现以下至少一种能力：因果链推理、边界意识、反事实分析、多条件综合判断、精确操作化、或对干扰信息的甄别。

# 可选进化策略（根据题目特点选择最合适的 1-3 种，不要全部堆砌）
1. **增加约束条件**：要求回答必须基于特定视角、法律条款、技术约束、场景限制或证据条件。
2. **要求显式推理**：不要只问"是什么"，而是问"为什么""如何排除其他可能""在什么条件下结论不成立"。
3. **引入边界或反事实**：改变原题中的某个条件，问结论会如何变化，或要求指出适用边界与例外。
4. **提升综合复杂度**：把两个相关子问题合并，要求比较、权衡、排序或推导优先级。
5. **抑制泛化回答**：要求回答必须紧扣本题具体情境，避免通用套话；或限制回答长度，要求"用最精炼的语言给出最关键的两点"。

# 原则
1. **可回答性**：新题必须仍能被强模型基于参考答案合理回答，不能引入需要外部未提供知识的隐藏条件。
2. **区分度**：新题应让"覆盖关键词但缺乏深度"的回答明显失分，让紧扣 reference 主线的回答得高分。
3. **不过拟合**：不要为了刁难某个候选答案而设置极窄陷阱；进化应提升题目本身质量，而不是针对特定措辞找茬。
4. **语言一致**：新题语言必须与原题一致。
5. **保持核心主题**：不要改变题目的领域和核心事实，只在深度、约束、推理要求上升级。

## 具体案例分析
### 1. `801`

- 轨迹：`1.00 -> 1.00 -> 1.00 -> 1.00 -> 0.95 -> 0.88`
- 题长：`50 -> 2194`

这一题从原始的“仅凭视频中嘴巴动了或手指点了一下试卷，在证据固定上有哪些风险点”，在 `round_1` 就被改造成了一个四层展开题：要求围绕“动作内容能否还原—是否存在合理替代解释—能否证明考生据此接收并改动了对应答案—哪些新增证据会显著改变判断”来回答。这个变化本身还是合理的，因为它把原题从泛泛列风险点，推进到了证明链拆解。

但从 `round_2` 开始，题目明显继续向“结构升级”推进，而不是向“单能力点边界测试”推进。例如：

- `round_2` 增加了“四步链条逐一判断”“最关键两个断点”“反事实场景重估”等要求；
- `round_3` 又把题目压成“4 个环节的最短证明链”，并要求找出“最致命的两个缺口”；
- `round_4` 进一步引入“甲/乙两种竞争性叙事”“3+1结构拆解”“X/Y/Z排序”等；
- `round_5` 最终变成“直接认定失败诊断”，要求先做 `A/B` 主轴二选一，再做 `X/Y/Z` 比较、`A/B/C/D` 路径诊断、最小补证等。

这题最后之所以从满分压到 `0.88`，不是因为它精准击中了某一个单点能力，而是因为它不断叠加：

- 主轴选择
- 证明顺序
- 多层结构拆解
- 多组对比项
- 失败路径诊断

Qwen 在 `round_5` 的主要失分点是“第一部分主轴选择与题目强制要求相悖”，其余部分仍然基本命中。这说明并不是核心事实判断错误，而是高复杂结构下的优先级选择偏差。

因此，这个样本得出的结论是：原始 prompt 确实能把题做到更难，但主要方式是不断增加结构和步骤，而不是更锋利地测试“哪一个事实节点是最关键缺口”。

### 2. `7337`

- 轨迹：`0.80 -> 0.96 -> 0.98 -> 1.00 -> 0.87 -> 0.97`
- 题长：`59 -> 1708`

原题本来只是“除了面貌和衣着，还需要结合哪些因素综合分析，以防跟丢目标”。到 `round_1` 时，它被改造成“地铁换乘站短暂脱视后，如何按优先级重新锁定目标，并指出哪些信息看似有用但不应作为主要依据”。把一般性知识题变成了现场策略题。

之后的几轮变化继续沿着“场景更具体、策略更复杂”的方向推进：

- `round_2` 加入了三个明确分流方向、三名侦查员的不同反馈，要求排序、保留双判据、给出弃跟标准；
- `round_3` 再增加“未来15秒只能下达一次主追和一次备份指令”的现场调度限制；
- `round_4` 与 `round_5` 继续加入“热饮/无回头”等新干扰信息、主追/备份/放后排序、口令设计和反事实窗口变化。

每轮都在加：

- 新场景细节
- 新干扰信息
- 新调度约束
- 新反事实比较

但得分轨迹却是 `0.80 -> 0.96 -> 0.98 -> 1.00 -> 0.87 -> 0.97`，说明 Qwen 几乎一直能处理这类复杂现场追踪策略题。即使 `round_4` 掉到 `0.87`，`round_5` 也又回到了 `0.97`。

因此，这个样本的结论是：把题目推成了一个越来越复杂的现场策略/调度题，但这种复杂化并没有稳定产生失败。Qwen 暴露的只是细微排序偏差，不是核心弱点。

### 3. `8638`

- 轨迹：`0.80 -> 1.00 -> 1.00 -> 1.00 -> 1.00 -> 1.00`
- 题长：`52 -> 2415`

该题为“复杂度膨胀但几乎没有区分收益”。原题只是：“为什么建议将图像保存为 PSD 而不是高压缩 JPG”。到 `round_1` 时，它已经变成了“JPG 作为唯一工作底图是否可行、JPG 在什么边界条件下可以使用、误存高压缩 JPG 在彩印和法庭质证中会放大什么风险”。

随后每一轮都继续加场景和约束：

- `round_2` 增加“反复增删箭头、比例尺、放大框、文字说明”和“三种用途：鉴定书正文/庭审展示/内部流转”；
- `round_3` 开始要求围绕“工作母版管理”做多项回答；
- `round_4` 再加入“与同一工作底图对应关系”“10分钟内切换三种版本”“领导/审签争议”等；
- `round_5` 最终形成“甲乙丙三种主张 + 六大任务模块 + 五维排序 + 受众区分”的长题。

题目从一个清楚的格式与母版边界问题，逐步变成了：

- 工作流管理题
- 多用途版本控制题
- 审签与出庭准备题
- 内外部受众沟通题

但 Qwen 全程都是满分，原始 prompt 没有创造新的判断难点，只是在不断扩展使用场景和组织结构。

### 4. `14865`

- 题长：`59 -> 1280`

这一题的原始问题为：“如何通过监控判断‘强行脱拽’猥亵，而非朋友间醉酒打闹”。`round_1` 时，它被改造成“先看什么、再排除什么、最后如何形成倾向性判断”，并要求给出两个最关键的支持特征和动摇条件。把题目从常识性判断推进到了链式研判。

但后续每轮仍然不断加结构：

- `round_2` 变成四步判定流程 + 三种解释排序 + 最希望补充的一条新事实；
- `round_3` 引入四个固定维度：动作方向一致性、接触部位/方式、第三人在场反应、接触结束后后果；
- `round_4` 进一步压缩成“最小判别链”，要求 4 个节点覆盖不同时间段，并说明每个节点如何联动；
- `round_5` 在此基础上又增加“删除单一维度后排序如何动摇”的扰动分析。

最终 Qwen 只在 `round_5` 掉到 `0.92`，且失分点是“删除单一维度时选错了最先动摇排序的维度”，而不是对“强行脱拽 vs 醉酒打闹”的主判断出错。

因此，这题的结论为：原始 prompt 的主要作用仍然是把题目推成多维结构化分析题，没有测出模型的弱点。

### 5. `18485`

- 轨迹：`0.83 -> 1.00 -> 0.99 -> 1.00 -> 1.00 -> 1.00`
- 题长：`61 -> 2378`

原题只是问：除了抓取伤人和抢枪瞬间，还应重点固定哪些关联画面。到 `round_1`，它被改造成按“案发前—案发中—案发后”列出优先固定画面，并说明各类画面的证明作用、缺失争议、不足条件，以及如何补强既遂/未遂判断。

但之后每轮都继续叠加：

- `round_2`：筛出“最关键的6类画面”，每类写 4 项，还要单列“如何排除先独立伤害、后临时起意接触配枪”的竞争解释；
- `round_3`：加入“二层判断流程”“跨层误判”“ABCD 情形比较”；
- `round_4`：要求每类画面固定写 6 项内容，并强调“先看哪一帧、排除什么、看不清就降到什么结论”；
- `round_5`：最终升级为“单类画面的证明上限 + 最小充分证明集合 + 反事实翻转点 + 排他论证”。

这里的变化体现为：

- 证明层级越来越多；
- 必要/充分区分越来越细；
- 竞争解释越来越完整；
- 输出结构越来越长。

但得分轨迹却是 `0.83 -> 1.00 -> 0.99 -> 1.00 -> 1.00 -> 1.00`。这说明对 Qwen 来说，这种“证明论结构加深”并没有形成新的边界，反而很可能因为它擅长结构化展开而持续高分。


## 添加约束规则
- 每次进化只能强化一个主能力：例如只强化“边界判断”或“反事实”，不能同时强化排序、表格、固定句式、反事实、误升级改写。
- 进化后题目必须保留一个清晰主轴，用一句话能说清“本题核心考什么”。
- 硬性输出模块最多 3 个，反事实变量最多 2-3 个，禁止6 个以上材料 + 6 个以上限制 + 多部分固定格式同时出现。
- 题目长度建议控制在原题的 3-8 倍，或设置绝对上限，比如 800-1200 中文字；除非原题本身很长。
- 禁止把难度主要建立在固定表格、固定句式、标签体系、枚举数量上。
- 必须输出 evaluation_focus：说明这道题最应该扣分的1-3 类错误，方便后续 rubric 聚焦。
- 必须输出 complexity_budget：列出新增了几个条件、几个反事实、几个输出任务，超过预算则不合格。
- 要求进化模型删除旧题中非必要的复杂约束，而不是只追加新约束。

### 1. `8638`：

轨迹：

- `0.90 -> 1.00 -> 1.00 -> 1.00 -> 1.00 -> 0.68`

题长：

- `52 -> 522`

原题只是问：“为什么建议保存为 PSD 而不是高压缩 JPG”。  
到 `round_1`，题目就被收束成一个非常单纯的判断：只围绕“保存方式本身是否会直接削弱图像中用于鉴定比对的关键细节”来判断 A/B 哪个更适合作为最终印刷源文件。这一步比原始 prompt 的做法要好很多，因为它没有立刻扩展到工作流、审签、出庭、版本切换等外部复杂场景，而是先把核心边界压缩出来。

接下来的几轮变化都沿着“边界更细、但主轴不变”的方式推进：

- `round_2` 要求答案必须先给结论，再用“因为……所以……”写出一条最短必要推理链；
- `round_3` 要求排除一类“看起来成立但在本题中不能作为依据的理由”；
- `round_4` 从“直接选 A/B”升级为“甲可写入最终结论，乙最多只能作为线索/不能作为本题结论依据”的二分边界判断；
- `round_5` 再增加 `丙` 这一条“更专业、更常用于工作流”的说法，迫使模型把“可写结论 / 线索 / 完全无关依据”三层真正区分开。

这题最终掉到 `0.68`，因为 `round_5` 的新增 `丙` 精确引入了一个更难的层级：  
Qwen 能判断 `甲` 与 `乙`，但没有把 `丙` 正确区分到“不可作为本题结论依据”，而是误判成“最多只能作为线索”。  
在这个样本上不是靠复杂度堆叠压分，而是靠**把“支持 B 的说法”进一步拆成“相关但不足”与“根本不构成结论依据”**来制造边界。

当前优化的prompt在这类题上开始形成有效边界，而且失分原因清晰、可归因。

### 2. `801`

轨迹：

- `1.00 -> 1.00 -> 1.00 -> 1.00 -> 0.80 -> 0.83`

题长：

- `50 -> 504`

原题只问“仅凭视频中嘴巴动了或手指点了一下试卷，在证据固定上存在哪些风险点”，属于开放式风险罗列题。

之后题目并没有像原始 prompt 那样迅速膨胀成多叙事、多排序、多证明链长题，而是逐步收束到“缺的究竟是哪一个关键环节”：

- `round_4` 进一步要求指出“如果要从线索升级为结论，还必须补上的那个最小关键事实是什么”；

- 直接锁定行为性质的事实；
- 只是结果层/旁证层的事实。

因此，这个样本说明：当前进化prompt在 `801` 上不只是“边界分类”，而是开始往“最小关键事实 / 最小充分条件”方向转。但它压出的错误还偏“原则提炼不够”。

3. 压分来得偏慢，前两轮几乎没有区分度，到后面才逐步显现。


# Question Evolution Pipeline

本目录包含一套用于 **PoliceQA 数据难度进化** 的脚本。核心目标：对当前候选模型（如 Qwen）得分过高的题目进行"题目进化"，使其能更好地区分强模型与弱模型，从而提升合成训练数据的质量。

> 背景：在 PoliceQA 上，rubric 增强遇到瓶颈——候选答案（Qwen）并不显著差于参考答案（GPT），差异多在风格偏好。因此我们把进化对象从 **rubric** 转向 **question**：让题目本身变得更难、更需要深度推理。

---

## 1. 整体流程

当前流水线准确流程是：

  1. 原始题目 + 原始 reference + 原始 rubric + 原始 score_prompt
     ↓
  2. Qwen 回答原题，得到 candidate_answer
     ↓
  3. Judge 按 rubric/score_prompt 给 Qwen 的 candidate_answer 打分
     当前默认 Judge 也是 Qwen
     ↓
  4. 挑选 Qwen 得分率 >= 0.8 的高分题
     ↓
  5. GPT-5.4 改写这些高分题，让题目更难
     ↓
  6. GPT-5.4 针对改写后的新题生成 reference
     ↓
  7. GPT-5.4 针对新题 + 新 reference 生成新 rubric 和新 score_prompt
     ↓
  8. Qwen 回答改写后的新题，得到新的 candidate_answer
     ↓
  9. Judge 再按新 rubric/score_prompt 给 Qwen 新答案打分

  所以每轮产物关系是：

  GPT 生成：
  - evolved_prompt
  - reference answer
  - rubric
  - score_prompt

  Qwen 生成：
  - 原题 candidate_answer
  - 新题 candidate_answer

  Judge 生成：
  - scoring_result

  当前代码里 Judge 默认也是 hjl_Qwen3.6-27B，所以可以理解
  成：

  Qwen 自己答题
  Qwen 按 rubric 给自己的答案打分
  GPT 负责把高分题升级并重建参考答案和评分标准
  Qwen 再答新题并评分

整个流水线包含 4 个脚本，执行顺序如下：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  阶段 0: 原始数据 (已包含 prompt + reference + rubric)                        │
│  data/police_qa_testset_v2.8.jsonl                                          │
└─────────────────────────────────┬───────────────────────────────────────────┘
                                  │
                                  ▼

│  输出:  *_scored.jsonl (新增 scoring_result)                                  │
└─────────────────────────────────┬───────────────────────────────────────────┘
                                  │
                                  ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  阶段 2: question_evolution.py  — 对高分题目进行进化（gpt-5.4）                │
│  输入:  *_scored.jsonl (含 scoring_result)                                    │
│  输出:  *_evolved.jsonl (prompt 被改写, 旧评分产物移入 meta_info.stale_*)      │
└─────────────────────────────────┬───────────────────────────────────────────┘
                                  │
                                  ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  阶段 3: collect_answers.py  — 为进化后的题目重新采集参考答案(gpt-5.4)          │

                                  │
                                  ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  阶段 4: gen_rubric.py  — 针对新题和新 reference 生成 rubric (gpt-5.4)         │

│  输出:  *_rubric.jsonl (新增 rubric, score_prompt)                            │
└─────────────────────────────────┬───────────────────────────────────────────┘
                                  │
                                  ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│  阶段 5: scoring.py  — 再次用候选模型(Qwen)答题并评分，验证进化效果              │
│  输入:  *_rubric.jsonl                                                        │
│  输出:  *_rubric_scored.jsonl (新增 scoring_result)                           │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.1 脚本职责速查

| 脚本 | 职责 | 典型输入 | 典型输出 |
| --- | --- | --- | --- |
| `scoring.py` | 调用候选模型生成答案，并用评分模型按 rubric 打分 | `*_rubric.jsonl` | `*_scored.jsonl` |
| `question_evolution.py` | 筛选高分题目，调用强模型把题目改得更难 | `*_scored.jsonl` | `*_evolved.jsonl` |
| `collect_answers.py` | 调用强模型为题目生成参考答案 | `*.jsonl` | `*_with_answers.jsonl` |
| `gen_rubric.py` | 根据题目和参考答案生成 rubric 与 score_prompt | `*_with_answers.jsonl` | `*_rubric.jsonl` |

## 2. 数据格式详解

数据采用 **JSONL** 格式，每行一个样本（一个 JSON 对象）。下文按流水线各阶段说明字段的"增删改"。

### 2.1 阶段 0：原始数据

以 `data/police_qa_testset_v2.8.jsonl` 为例，每行至少包含：

```json
{
  "index": 11112,
  "prompt": "在使用重合比较法（将两台摄像机画面重叠或将照片重叠比对）来研判嫌疑人身份时，对两张照片的拍摄角度有什么具体要求？",
  "meta_info": {
    "references": ["参考答案文本，来自 GPT-5.4"],
    "answer_from_book": "教材/资料中的简版答案",
    "source_file": "视频侦查技术-公大社-2015.jsonl",
    "labels": { "topic": ["情报研判"], "difficulty": "专业", ... },
    ...
  },
  "rubric": [
    { "title": "核心要点", "description": "...", "weight": 4 },
    { "title": "边界条件", "description": "...", "weight": 3 },
    { "title": "常见错误", "description": "...", "weight": -2 }
  ],
  "rubric_thought_process": "rubric 设计思路...",
  "score_prompt": "你是严格的模型评测打分员...<<<待评答案>>..."
}
```

#### 字段说明

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `index` | int | 样本编号 |
| `prompt` | string | **题目文本**，是模型需要回答的问题 |
| `meta_info` | object | 元数据容器 |
| `meta_info.references` | list[string] | 参考答案列表，通常取 `[0]` 作为标准答案；由 GPT-5.4 生成 |
| `meta_info.answer_from_book` | string | 教材/资料原始答案，可作参考 |
| `meta_info.labels` | object | 题目标签：主题、难度、题型、场景等 |
| `rubric` | list[object] | 评分标准；`weight > 0` 为加分项，`weight < 0` 为扣分项 |
| `rubric[].title` | string | 评分维度标题 |
| `rubric[].description` | string | 评分细则 |
| `rubric[].weight` | int | 该项满分/扣分值 |
| `rubric_thought_process` | string | 生成 rubric 时的设计思路 |
| `score_prompt` | string | 给评分模型的完整提示词，其中 `<<<待评答案>>>` 为占位符 |

---

### 2.2 阶段 1：`scoring.py` 输出

`scoring.py` 会在每条样本上新增一个 `scoring_result` 字段，记录候选模型（如 Qwen）的回答及评分结果。

```json
{
  "index": 11112,
  "prompt": "...",
  "meta_info": { "references": [...], ... },
  "rubric": [...],
  "score_prompt": "...",
  "scoring_result": {
    "answer_mode": "llm",
    "answer_model": "hjl_Qwen3.6-27B",
    "candidate_answer": "候选模型的完整回答文本...",
    "item_scores": [
      { "title": "核心要点", "weight": 4, "awarded": 4, "brief_reason": "覆盖了关键信息" },
      { "title": "边界条件", "weight": 3, "awarded": 2, "brief_reason": "提到了部分边界" },
      { "title": "常见错误", "weight": -2, "awarded": 0, "brief_reason": "未触发" }
    ],
    "overall_comment": "整体评价...",
    "total_awarded": 6,
    "total_possible": 7,
    "judge_model": "hjl_Qwen3.6-27B",
    "judge_raw_response": "评分模型的原始输出..."
  }
}
```

#### 字段说明

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `scoring_result` | object | 评分结果容器 |
| `scoring_result.answer_mode` | string | `reference`（用参考答案自评）或 `llm`（调用候选模型） |
| `scoring_result.answer_model` | string | 生成 candidate_answer 的模型名 |
| `scoring_result.candidate_answer` | string | 候选模型对 `prompt` 的回答 |
| `scoring_result.item_scores` | list[object] | 逐条 rubric 得分；`title` 必须与 `rubric` 严格一致 |
| `scoring_result.total_awarded` | int | 实际总得分（含负分项扣分） |
| `scoring_result.total_possible` | int | 正分项满分之和 |
| `scoring_result.judge_model` | string | 执行评分的模型 |
| `scoring_result.judge_raw_response` | string | 评分模型原始返回，用于 debug |

#### 得分率计算

```text
score_rate = scoring_result.total_awarded / scoring_result.total_possible
```

`question_evolution.py` 默认对 `score_rate >= 0.8` 的题目进行进化。

---

### 2.3 阶段 2：`question_evolution.py` 输出

这是本套程序的核心。它会：

1. **全量输出**所有样本；
2. 对 `score_rate >= --min-score-rate` 的样本：
   - 把原 `prompt` 移到 `meta_info.prompt_old`；
   - 把 `prompt` 替换为模型生成的 `evolved_prompt`；
   - 把原 `rubric` / `score_prompt` / `scoring_result` 移到 `meta_info.stale_*`；
   - 新增 `meta_info.question_evolution_metadata`；
   - 顶层新增 `question_evolved: true`。
3. 对未触发进化的样本：
   - 原样透传；
   - 顶层新增 `question_evolved: false`。

> 为什么要把 rubric/score_prompt/scoring_result 移走？因为 `prompt` 变了，旧的 rubric 和评分结果已经失效，必须重新生成。

#### 进化后的样本示例

```json
{
  "index": 11112,
  "prompt": "升级后的新题目文本...",
  "question_evolved": true,
  "meta_info": {
    "references": ["原参考答案，此时已不适用新题"],
    "prompt_old": "原题文本...",
    "stale_rubric": [
      { "title": "核心要点", "description": "...", "weight": 4 }
    ],
    "stale_score_prompt": "旧的评分提示词...",
    "stale_scoring_result": {
      "total_awarded": 6,
      "total_possible": 7,
      ...
    },
    "question_evolution_metadata": {
      "question_evolved": true,
      "trigger_score_rate": 0.857,
      "question_evolution_model": "gpt-5.4",
      "evolution_strategy": "增加反事实条件与最小充分证据要求...",
      "notes_for_reference": "基本适用",
      "question_evolution_raw_response": "模型原始返回..."
    },
    ...
  }
}
```

#### 字段说明

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `prompt` | string | **进化后的新题目**；后续所有步骤都基于这个 prompt |
| `question_evolved` | bool | 本题是否被进化 |
| `meta_info.prompt_old` | string | 原题文本，便于对比 |
| `meta_info.stale_rubric` | list[object] | 旧 rubric（对新题已失效，仅供参考） |
| `meta_info.stale_score_prompt` | string | 旧 score_prompt（对新题已失效） |
| `meta_info.stale_scoring_result` | object | 旧评分结果（对新题已失效） |
| `meta_info.question_evolution_metadata` | object | 进化元数据 |
| `meta_info.question_evolution_metadata.trigger_score_rate` | float | 触发进化的得分率 |
| `meta_info.question_evolution_metadata.evolution_strategy` | string | 采用的进化策略说明 |
| `meta_info.question_evolution_metadata.notes_for_reference` | string | 原参考答案是否仍适用 |

---

### 2.4 阶段 3：`collect_answers.py` 输出

`collect_answers.py` 会调用强模型（默认 GPT-5.4）为每个 `prompt` 生成参考答案，并覆盖 `meta_info.references`。

输出结构（仅保留关键字段）：

```json
{
  "index": 11112,
  "prompt": "升级后的新题目文本...",
  "meta_info": {
    "references": ["新的参考答案，由 GPT-5.4 针对 evolved_prompt 生成"],
    "prompt_old": "原题文本...",
    "stale_rubric": [...],
    "stale_score_prompt": "...",
    "stale_scoring_result": {...},
    "question_evolution_metadata": {...},
    ...
  }
}
```

注意：`collect_answers.py` 的输出只保留 `index`、`prompt`、`meta_info` 三个顶层字段。原来的顶层 `rubric`、`score_prompt`、`scoring_result`、`question_evolved` 会被丢弃（但 `meta_info` 里的历史信息会保留）。

---

### 2.5 阶段 4：`gen_rubric.py` 输出

`gen_rubric.py` 读取 `meta_info.references[0]` 作为参考答案，为新的 `prompt` 生成 rubric。

输出会在阶段 3 的基础上新增：

```json
{
  "index": 11112,
  "prompt": "升级后的新题目文本...",
  "meta_info": { "references": [...], "prompt_old": "...", ... },
  "rubric": [
    { "title": "...", "description": "...", "weight": 5 }
  ],
  "rubric_thought_process": "...",
  "score_prompt": "...<<<待评答案>>..."
}
```

---

### 2.6 阶段 5：第二次 `scoring.py` 输出

与阶段 1 相同，再次用候选模型（Qwen）回答新题，并用评分模型打分。

```json
{
  "index": 11112,
  "prompt": "升级后的新题目文本...",
  "meta_info": { "references": [...], "prompt_old": "...", ... },
  "rubric": [...],
  "score_prompt": "...",
  "scoring_result": {
    "answer_mode": "llm",
    "answer_model": "hjl_Qwen3.6-27B",
    "candidate_answer": "候选模型对新题的回答...",
    "item_scores": [...],
    "total_awarded": 4,
    "total_possible": 10,
    ...
  }
}
```

通过对比 `meta_info.stale_scoring_result` 和新的 `scoring_result`，即可判断题目进化是否有效：理想情况下，新题的候选模型得分率应显著下降。

## 4. 快速开始

### 4.1 单轮运行

```bash
bash run.sh
```

`run.sh` 中已经写好了单轮完整流程。阶段 1（scoring）和阶段 2（question_evolution）默认已启用；阶段 0（原始参考答案采集）和阶段 2 之前的前置 rubric 生成因为输入数据 `police_qa_testset_v2.8.jsonl` 已包含 `references` 和 `rubric`，所以被注释掉。

### 4.2 多轮循环运行（推荐）

如果你想让 question evolution 自动循环多轮，直到 Qwen 平均得分率低于阈值或达到最大轮数，使用：

```bash
bash run_loop.sh
```

默认配置：

- 最大轮数：`MAX_ROUNDS=20`
- 提前停止阈值：`EARLY_STOP_RATE=0.5`（当某轮 Qwen 平均得分率 < 50% 时停止）
- 每轮触发进化的阈值：`MIN_SCORE_RATE=0.8`

每轮结果保存在 `exp/round_N/` 子文件夹中：

```text
exp/
├── round_0/
│   ├── input.jsonl       # 初始输入（从 data/police_qa_testset_v2.8.jsonl 复制）
│   └── scored.jsonl      # 初始 baseline 评分结果
├── round_1/
│   ├── input.jsonl       # 复制自 round_0/scored.jsonl
│   ├── evolved.jsonl     # question_evolution 输出
│   ├── with_answers.jsonl
│   ├── rubric.jsonl
│   └── scored.jsonl      # 本轮复测结果
├── round_2/
│   └── ...
├── summary.txt           # 各轮平均得分率汇总
└── final/
    └── final_scored.jsonl
```

`run_loop.sh` 会自动生成 `exp/summary.txt`，方便你观察得分率下降趋势：

```text
Round | Avg Score Rate | Status
------|----------------|--------
    0 |         0.7865 | baseline
    1 |         0.6123 | continue
    2 |         0.4532 | early_stop
```

脚本也支持**断点续跑**：如果某轮的 `scored.jsonl` 已存在且非空，则跳过该轮执行，直接读取已有结果进行评估。因此即使中途中断，也可以直接重新运行 `bash run_loop.sh` 继续。

#### 修改循环参数

直接编辑 `run_loop.sh` 顶部的配置区即可：

```bash
MAX_ROUNDS=20
EARLY_STOP_RATE=0.5
MIN_SCORE_RATE=0.8
```

## 5. 常见问题

### Q1：如何只看哪些题目被进化了？

阶段 2 的输出中，过滤 `question_evolved == true`：

```python
import json

with open("data/questions_evolved.jsonl", "r", encoding="utf-8") as f:
    for line in f:
        item = json.loads(line)
        if item.get("question_evolved"):
            print(item["meta_info"]["prompt_old"])
            print("→", item["prompt"])
            print()
```

### Q2：阶段 3 之后如何知道哪些题是进化过的？

阶段 3 会丢弃顶层 `question_evolved`，但保留了 `meta_info.question_evolution_metadata`：

```python
if item["meta_info"].get("question_evolution_metadata", {}).get("question_evolved"):
    print("本题是进化题")
```

### Q3：为什么进化后要把 rubric 移走而不是直接更新？

题目进化后，问题的核心、约束、推理要求都可能改变，旧的 rubric 可能不再匹配新题。因此本程序选择把旧 rubric 标记为 `stale_`，让 `gen_rubric.py` 针对新 prompt 和新 reference 重新生成 rubric，避免评分标准与新题脱节。

### Q4：`--min-score-rate` 设多少合适？

默认值 `0.8` 是一个经验值：得分率 80% 以上说明候选模型（Qwen）基本答对了这道题，题目对当前候选模型区分度不足。你可以根据实际得分分布调整：

- 想进化更多题：降到 `0.7`
- 只想进化满分题：提高到 `0.9` 或 `1.0`

### Q5：可以只对进化题重新采集参考答案吗？

可以。先用 `jq` 或 Python 过滤出 `question_evolved == true` 的样本，再喂给 `collect_answers.py`：

```bash
python -c "
import sys, json
for line in sys.stdin:
    j = json.loads(line)
    if j.get('question_evolved'):
        print(json.dumps(j, ensure_ascii=False))
" < data/questions_evolved.jsonl > data/questions_evolved_only.jsonl

python collect_answers.py --input data/questions_evolved_only.jsonl --output data/questions_evolved_only_with_answers.jsonl ...
```

## 7. 调试建议

1. **先看少量样本**：不要直接跑整个数据集。先用 `head -n 5` 切一个小文件验证流程。
2. **检查进化质量**：重点看 `meta_info.prompt_old` → `prompt` 的变化是否合理，是否引入了外部未提供的知识。
3. **对比前后得分**：通过 `meta_info.stale_scoring_result.total_awarded/total_possible` 与新 `scoring_result` 对比，判断进化是否有效。
4. **查看失败文件**：每个脚本失败的数据会写入 `*.failed` 文件，里面包含错误信息。
