# Question Evolution Pipeline 项目设计文档

## 1. 文档目的

本文档用于明确 `question_evolution_pipeline` 的后续设计方向，目标不是把题目机械地“越改越难”，而是稳定构建一套能够：

1. 从原始 PoliceQA 数据中筛出真正值得进化的样本；
2. 针对不同类型样本选择合适的 question evolution 策略；
3. 让进化后的题目稳定暴露候选模型的真实能力边界；
4. 将有效经验沉淀为可复用的机制，而不是停留在单轮实验结论中。

本文档覆盖：

1. 样本筛选标准
2. 题目画像 schema
3. 进化算子设计与路由机制
4. 评估指标与停止条件
5. 经验复用与实验沉淀机制
6. 当前项目还需要补强的关键点

---

## 2. 项目目标与非目标

### 2.1 核心目标

项目的核心目标是：通过多轮 `question evolution`，定向压测候选模型在警务问答、证据判断、边界推理、时空关联、程序规范类题目上的真实短板，并沉淀出高价值训练样本。

更具体地说，理想的进化结果应满足：

1. 新题仍然考查原题的核心领域与核心能力；
2. 新题只有一个清晰主轴；
3. 候选模型主要在一个可解释的错误点上失分；
4. rubric 能稳定捕捉这个错误，而不是靠篇幅、格式或术语密度打分；
5. 进化后的失败可以归因为具体能力缺口，而不是“题太长”或“规则太绕”。

### 2.2 非目标

以下方向不应被当作项目成功：

1. 单纯把题目写得更长、更复杂；
2. 让平均分下降，但无法解释为什么下降；
3. 通过增加格式要求、编号要求、表格要求压分；
4. 通过引入题干外知识制造不可回答样本；
5. 只追求生成数量，而不管样本价值和边界清晰度。

---

## 3. 当前项目的主要问题

结合现有实现、实验分析和样本观察，当前项目主要存在以下问题：

### 3.1 高分样本不等于高价值进化样本

当前流程主要基于 `score_rate >= threshold` 触发进化，但高分题中混有多类样本：

1. 本身就容易判断、区分度天然低的题；
2. 纯定义或纯流程回忆题；
3. 候选答案真的具备扎实能力，因此高分合理；
4. 候选答案只是靠套话、泛化、层级越推获得虚高分。

真正值得进入 pipeline 的，是第 4 类以及部分边界不清的第 3 类，而不是全部高分题。

### 3.2 一个 evolution prompt 无法覆盖所有题型

不同样本的“虚高主因”并不相同：

1. 有的题是泛化罗列；
2. 有的题是把线索直接升级为结论；
3. 有的题是多个缺口都像关键点，但没抓准最关键的一个；
4. 有的题是引入了题干外前提；
5. 有的题是显眼动作抓住了，但漏掉真正决定定性的那一层。

因此不能依赖一个统一策略覆盖所有样本，必须做算子分类与路由。

### 3.3 复杂度控制主要停留在 prompt 层

当前设计已经强调“单主轴”“复杂度预算”“避免重复题型”，这是正确方向。但如果没有代码层校验，多轮链式进化后仍然容易出现：

1. 题长膨胀；
2. 任务数膨胀；
3. 反事实堆叠；
4. 隐性重复问法；
5. rubric 与题面共同膨胀，导致分数失真。

### 3.4 经验沉淀还不够结构化

目前大量有效经验散落在实验分析文档中，但还没有形成统一的、可检索的经验库。结果是：

1. 同类样本可能重复踩同样的失败路径；
2. 某种进化策略在什么样本上有效，难以量化；
3. prompt 优化更多依赖人工印象，而不是可归纳证据。

### 3.5 Judge 与 rubric 仍存在归因风险

当前项目默认使用 Qwen 进行回答与评分，虽然流程上可跑通，但要持续警惕：

1. 自评偏好；
2. rubric 漂移；
3. 进化后分数变化来自 judge 风格，而非真实区分度提升；
4. 未进化样本与进化样本的统计口径混用。

## 4. 设计原则

后续所有设计应遵守以下原则：

### 4.1 单主轴原则

每次进化只针对一个核心能力缺口，不在同一题上同时压多个错误。

### 4.2 可归因原则

理想的失分应能明确回答：

1. 题目新增了什么要求；
2. 候选模型错在什么地方；
3. rubric 为什么能稳定扣到这里。

### 4.3 复杂度预算原则

难度增加不等于信息和任务无限叠加。每轮进化都必须有明确预算。

### 4.4 样本分型原则

不同类型样本应走不同进化算子，不做“一套 prompt 打天下”。

### 4.5 先筛选、后进化原则

不值得进化的原题应尽早剔除，减少无效算力消耗与噪声。

### 4.6 经验复用原则

进化成功或失败都要沉淀为结构化经验，用于后续算子选择、prompt 调整与样本路由。

## 5. 样本筛选设计

样本筛选分两层进行。

### 5.1 第一层：原始数据准入筛选

目标是从原始数据里先筛掉明显不适合做 question evolution 的样本。

#### 5.1.1 优先纳入的样本类型

优先纳入以下题目：

1. 需要多步推理而非单点回忆；
2. 需要区分“线索/可疑/待核查/可写结论”等不同证据层级；
3. 需要处理监控盲区、时间差、跨摄像头轨迹拼接、身份排他性判断；
4. 需要比较两个都像关键点的缺口；
5. 需要程序规范、证据固定、合法性边界判断；
6. 需要在题面事实范围内完成排除性推理。

#### 5.1.2 应排除的样本类型

优先排除以下题目：

1. 一眼就能定性或定罪的题；
2. 纯概念定义题；
3. 单纯背流程即可高分的题；
4. 主要考记忆，而不是推理或判断的题；
5. 强依赖题外专业知识才能回答的题；
6. 题面信息严重不足，天然容易变成“无法判断”的题。

### 5.2 第二层：进入进化前的高分样本筛选

对通过第一层筛选的样本，再结合第一轮 scoring 结果决定是否进入进化。

#### 5.2.1 建议进入进化的条件

建议同时满足：

1. `score_rate >= threshold`
2. 候选答案存在虚高迹象
3. 题目具有进一步聚焦主轴的空间
4. 原参考答案或稍作扩展后仍能支持新题

#### 5.2.2 典型虚高迹象

可作为进入进化的强信号：

1. 候选答案泛泛罗列，缺少题面事实绑定；
2. 候选答案直接给结论，但中间链条缺失；
3. 候选答案把线索上升为定性结论；
4. 候选答案说“还缺证据”，但说不准最关键缺口；
5. 候选答案抓住显眼点，但漏掉真正卡结论的那一层。

#### 5.2.3 不建议继续进化的高分样本

以下情况即使高分，也可直接透传或退出：

1. 题目本身价值不高；
2. 候选答案高分是合理高分；
3. 题目已足够聚焦，但模型确实掌握得好；
4. 再进化极易引入题外知识或复杂度膨胀。

## 6. 题目画像 Schema 设计

为了支持筛选、路由、分析和经验复用，需要为每个样本增加结构化画像。建议在现有 `meta_info.labels` 基础上扩展如下字段。

### 6.1 基础画像字段

1. `core_capability`
   - 概念识别
   - 证据链补强
   - 时空关联
   - 边界判断
   - 排他性认定
   - 反事实推理
   - 程序规范
   - 行为模式识别

2. `claim_level`
   - 事实识别
   - 可疑线索
   - 高度怀疑
   - 可写结论
   - 程序合法性判断

3. `problem_shape`
   - 单概念
   - 双概念比较
   - 多条件组合
   - 多阶段流程
   - 候选项区分

4. `reasoning_granularity`
   - 单步判断
   - 两步链条
   - 多步链条

5. `answer_mode_expected`
   - 罗列型
   - 比较型
   - 排除型
   - 排序型
   - 选择型

### 6.2 风险画像字段

1. `easy_judgment_risk`
   - high / medium / low

2. `external_knowledge_risk`
   - high / medium / low

3. `complexity_expansion_risk`
   - high / medium / low

4. `rubric_drift_risk`
   - high / medium / low

### 6.3 候选答案虚高画像字段

1. `candidate_overscore_cause`
   - 泛化罗列
   - 层级越推
   - 题外补设
   - 漏最小关键事实
   - 抓显眼点漏关键层
   - 受干扰信息带偏

2. `target_failure_mode`
   - 选错最关键缺口
   - 误把线索写成结论
   - 无法分层相近说法
   - 不能绑定题干事实
   - 引入题外前提

### 6.4 画像用途

这些画像字段将用于：

1. 样本准入筛选
2. evolution operator 路由
3. rubric 设计聚焦
4. 实验分组统计
5. 经验检索与复用

## 7. 进化算子设计

### 7.1 总体思路

不把 question evolution 视为“改写题目”这一种操作，而是拆解为多个可选算子。每轮只选择一个主算子，必要时允许一个轻量辅助算子。

### 7.2 推荐的核心算子

#### O1. 候选缺口二选一

适用样本：

1. 存在两个都像关键点的缺口；
2. 直接问“还缺什么”容易产生多个合理答案。

目标：

1. 迫使模型比较两个近似缺口；
2. 暴露其是否抓准“真正最卡结论”的那一条。

#### O2. 子判断定位

适用样本：

1. 结论可以拆成多个子判断；
2. 候选答案经常跳过中间层级。

目标：

1. 先定位到底是哪一个子判断还不能成立；
2. 再要求说明最少还缺什么。

#### O3. 单步跳跃识别

适用样本：

1. 候选答案经常从线索直接跳到结论；
2. 关键问题在“跳了哪一步”，而不是“答得不够长”。

目标：

1. 让模型指出具体是从哪一步跳到哪一步；
2. 压测证据层级控制能力。

#### O4. 近似项分层

适用样本：

1. 多个说法都不能直接写结论；
2. 真正难点在于继续分层，而不是做可写/不可写二分。

目标：

1. 暴露模型对相近证据层级的精细区分能力；
2. 避免候选答案只会笼统说“证据不足”。

#### O5. 题干外补设识别

适用样本：

1. 候选答案经常偷偷引入题外前提；
2. 原题事实边界比较清晰。

目标：

1. 检查模型能否识别哪些判断依赖了题外信息；
2. 压测事实约束意识。

#### O6. 单变量反事实

适用样本：

1. 原题已有明确主轴；
2. 只改一个条件就能改变判断层级。

目标：

1. 检查模型是否理解哪个条件真正决定结论；
2. 避免堆叠多组反事实。

#### O7. 具体化约束

适用样本：

1. 候选答案高度模板化；
2. 题目容易被“行业套话”覆盖。

目标：

1. 强制答案绑定题干事实；
2. 压制泛化回答。

### 7.3 算子禁忌

以下做法不应视为有效算子：

1. 只把“最小关键事实/最小前提/最小跳步”换壳复用；
2. 一次性叠加多个算子导致主轴混乱；
3. 通过增加长格式要求压分；
4. 通过题外知识或隐性规则制造陷阱；
5. 为了打击某个答案措辞而设计极窄问题。

## 8. 进化机制设计

### 8.1 总体机制

建议采用“链式主流程 + 局部树状探索”的混合机制。

### 8.2 链式主流程

链式仍然是项目的主运行方式，因为最终需要为每个样本沉淀一条可复用的演化轨迹。

基本流程：

1. 原题 scoring
2. 样本画像生成
3. operator 路由
4. 生成 1 个主进化版本
5. 新题 reference/rubric/scoring
6. 判断是否继续下一轮

### 8.3 局部树状探索

对高价值样本，在某一轮不要只生成 1 个 evolved prompt，而是并行生成 2-4 个候选版本：

1. 使用不同主算子；
2. 或在同一算子下生成不同表达实现。

然后基于以下标准选 1 个进入主链：

1. 主轴是否更清晰；
2. 是否更可回答；
3. 是否更容易稳定评分；
4. 是否更可能压出单一错误；
5. 是否控制在复杂度预算内。

### 8.4 为什么不用纯树状

纯树状虽然探索能力强，但当前项目更需要：

1. 过程可解释；
2. 成本可控；
3. 样本轨迹可复盘；
4. 经验可沉淀。

因此树状探索只适合用于高价值难样本，不建议全量展开。

## 9. 进化路由机制

### 9.1 路由输入

路由决策应基于：

1. 样本画像
2. 候选答案虚高主因
3. 上一轮进化方式
4. 上一轮失分是否成功集中
5. 复杂度预算剩余空间

### 9.2 路由规则示例

1. `candidate_overscore_cause = 漏最小关键事实`
   - 优先 `O1 候选缺口二选一`
   - 次选 `O2 子判断定位`

2. `candidate_overscore_cause = 层级越推`
   - 优先 `O3 单步跳跃识别`
   - 次选 `O4 近似项分层`

3. `candidate_overscore_cause = 题外补设`
   - 优先 `O5 题干外补设识别`

4. `candidate_overscore_cause = 泛化罗列`
   - 优先 `O7 具体化约束`

5. `claim_level` 对条件敏感
   - 优先 `O6 单变量反事实`

### 9.3 去重复约束

针对同一样本：

1. 若上一轮已用 `O1` 且效果一般，本轮不要只换壳继续问“还缺哪一条”；
2. 若上一轮已是边界分类题，本轮不再重复做同结构三分类；
3. 若题目已变长但仍高分，本轮优先收束主轴，而不是继续加条件。

## 10. 复杂度预算与验证

### 10.1 预算维度

建议对每轮 evolved question 记录以下预算字段：

1. `main_axis_count`
2. `new_facts_count`
3. `output_tasks_count`
4. `candidate_options_count`
5. `counterfactual_count`
6. `hard_constraints_count`
7. `estimated_prompt_chars`

### 10.2 推荐预算上限

默认建议：

1. 主轴数必须为 1；
2. 新增事实最多 3 条；
3. 输出任务最多 2 个，优先 1 个；
4. 候选项最多 3 个；
5. 反事实最多 1 组；
6. 若上一轮题目已偏长，本轮优先减法而不是加法。

### 10.3 代码级验证

建议新增 `validate_evolved_question()` 之类的验证逻辑，对以下情况直接拒绝：

1. 主轴不清；
2. 多任务堆叠；
3. 长度超预算；
4. 出现明显题外知识依赖；
5. 与上一轮属于重复题型；
6. 题面不具备稳定评分条件。

## 11. 评分与评估设计

### 11.1 不应只看平均分

项目的核心评估对象不是“平均分是否下降”，而是“是否压出了更清晰的能力边界”。

### 11.2 核心评估指标

建议至少维护以下指标：

1. `score_rate_before`
2. `score_rate_after`
3. `delta_score_rate`
4. `full_score_count`
5. `question_length`
6. `evolved_count`
7. `judge_agreement_rate`
8. `rubric_stability_flag`
9. `answer_error_concentration`

### 11.3 关键分析维度

每个样本至少要回答：

1. 题面到底怎么变了；
2. 失分是否集中在一个错误；
3. 这个错误是否是题面变化直接压出来的；
4. 这是有效压测，还是复杂度膨胀；
5. 是否出现不可回答或多解风险。

### 11.4 结果分层

每轮结果建议分为三类：

1. `有效边界样本`
   - 失分原因明确
   - 和题面变化直接对应
   - 可沉淀为高价值样本

2. `无效复杂样本`
   - 题变长变复杂
   - 但没有稳定压分或失分不可解释

3. `不可回答/陷阱样本`
   - 题面事实不足
   - 正确行为接近“指出不能判断”
   - 不适合作为稳定评测样本

---

## 12. 停止条件设计

同一个样本不应无限进化。建议建立 stop rule。

### 12.1 建议停止进化的情况

1. 已经稳定压出单一、可解释的失分；
2. 连续两轮没有带来更清晰的区分收益；
3. 题目复杂度持续上升但分数基本不变；
4. 出现不可回答风险；
5. rubric 已难以稳定跟住题面；
6. 样本已偏离原题核心能力。

### 12.2 样本终态分类

每个样本最终应进入下列之一：

1. `保留为高价值进化样本`
2. `退回原题或较早轮版本`
3. `标记为无效复杂化样本`
4. `标记为不适合继续进化`

---

## 13. 经验复用机制设计

### 13.1 目标

把“某类样本适合什么进化方向”从人工经验变成结构化资产。

### 13.2 经验单元设计

建议每次有效或失败进化都记录一条经验单元，至少包括：

1. `sample_signature`
   - 核心能力类型
   - 结论层级
   - 问题结构

2. `candidate_failure_mode`
   - 候选答案虚高主因
   - 最终暴露出的真实错误

3. `operator_used`
   - 使用的主算子
   - 是否使用辅助算子

4. `effect_result`
   - 是否成功降分
   - 是否成功集中失分
   - 是否造成不可回答
   - 是否造成复杂度膨胀

5. `notes`
   - 对后续相似样本的建议

### 13.3 经验库的用途

经验库至少服务于三件事：

1. 进化前检索相似样本并推荐算子；
2. prompt 迭代时分析哪些策略真实有效；
3. 形成“成功模式”和“失败模式”双库。

### 13.4 成功库与失败库都要保留

失败经验同样重要。尤其要沉淀以下失败模式：

1. 题目越改越长但没有收益；
2. 反事实加了，但核心能力没变；
3. 变成多解题；
4. 过度依赖 rubric 暗规则；
5. 题型表面变化了，本质仍在重复上一轮。

---

## 14. Judge 与 Rubric 控制建议

### 14.1 Judge 风险控制

当前流程中，judge 与 answer model 存在重合时，建议增加以下控制：

1. 抽样双 judge 复核；
2. 对关键实验做 GPT judge 对照；
3. 单独分析 Qwen judge 与 GPT judge 的偏差；
4. 不将 judge 风格差异误认成 question evolution 成效。

### 14.2 Rubric 风险控制

需持续检查：

1. rubric 是否跟着题面一起过度膨胀；
2. rubric 是否捕捉了题面真正主轴；
3. rubric 是否把格式完整性当作主要得分来源；
4. 未进化样本是否被无意重写 rubric。

## 15. 能力边界的类型划分与优化决策框架

### 15.1 为什么需要单独讨论“边界来源”

本项目的核心目标是挖出 `Qwen3.6-27B` 的能力边界，但“发现边界”本身不是终点。真正关键的是继续回答：

1. 这类边界主要是训练分布问题，还是模型容量问题；
2. 这类边界是否适合通过补数据和微调改善；
3. 这类边界是否更适合通过更大模型、推理拆解或系统路由解决。

如果不做这一步区分，项目很容易退化成：

1. 找到很多边界题；
2. 把这些题回灌训练；
3. 在相似题上略有提升；
4. 但对更广泛任务泛化有限，甚至误以为“边界被解决了”。

因此，项目输出不应只是“哪些题压出了 Qwen 的短板”，还应进一步回答“这些短板中，哪些是可训练边界，哪些是容量边界，哪些应交给系统设计解决”。

### 15.2 能力边界不等于都由参数量决定

模型在某类题上失分，可能来自多种原因，而不应默认都由参数量决定：

1. `纯容量上限`
   模型难以稳定维持长链推理、中间状态或细粒度多层比较。

2. `训练分布覆盖不足`
   模型不是完全不会，而是没见过足够多这一类任务结构。

3. `调用不稳定`
   模型具备部分能力，但在复杂题上经常调不出来，表现为会说“证据不足”，但说不准最关键缺口是哪一个。

4. `回答习惯偏差`
   模型倾向于泛化罗列、题外补设、把线索升级为结论。这类问题更像行为模式偏差，而不是纯参数不足。

因此，同一个“能力边界”表面上看都表现为失分，但其优化路径可能完全不同。

### 15.3 三类边界

建议把本项目挖到的边界至少分成三类。

#### A. 可训练边界（Trainable Boundary）

这类边界通常具有以下特征：

1. 错误模式稳定同构；
2. 失分点清晰集中；
3. 通过结构化提示、拆步回答或少量示例即可明显改善；
4. 更像“没学会这类判断方式”，而不是“根本装不下”。

常见例子：

1. 能指出“还缺证据”，但说不准最少还缺哪一条；
2. 容易把“可疑线索”写成“可下结论事实”；
3. 面对两个都像关键点的缺口时，选错真正最关键的那个；
4. 会泛化罗列，但不能紧扣题干事实。

对这类边界，数据和训练通常是有效优化手段。

#### B. 规模敏感边界（Scale-sensitive Boundary）

这类边界通常具有以下特征：

1. 随模型参数规模扩大而明显改善；
2. 需要更稳定地维持多层比较或长链中间状态；
3. 小模型即使理解方向对，也容易在关键层级比较时失稳。

常见例子：

1. 多个近似证据层级的细粒度排序；
2. 长链排除性推理中的全局一致性维持；
3. 同时比较多个解释路径时的主轴稳定性。

对这类边界，补数据可能有局部收益，但通常不能指望仅靠微调完全解决。

#### C. 系统级边界（System-level Boundary）

这类边界不一定是模型本体完全不会，而是更适合通过外部流程或推理机制补救。

常见例子：

1. 先判定证据层级，再生成最终结论会显著更稳；
2. 先拆成子判断，再做“最小关键缺口”比较会显著更稳；
3. 高风险边界题应路由给更强模型，而不是强求 27B 单独解决。

对这类边界，更适合优化：

1. prompt 结构；
2. 推理流程；
3. 分步判别；
4. 模型路由；
5. 外部工具或结构化中间层。

### 15.4 训练目标不应是“补题”，而应是“补失败机制”

即使一个边界属于可训练边界，也不应简单把边界题本身回灌训练。原因是：

1. 直接回灌原题，容易变成补 benchmark；
2. 对近邻样本可能有提升；
3. 但对更广泛任务的泛化通常不足。

更合理的训练路径应是：

`边界样本 -> 失败模式 -> 任务族 -> 训练分布`

也就是说，真正进入训练集的，不应只是某几道边界题，而应是这类边界背后的“能力族”。

例如：

1. 线索 / 嫌疑 / 待核查 / 可写结论的层级区分；
2. 最小关键缺口的判断；
3. 两个近似缺口的二选一比较；
4. 在题干内事实约束下完成排除性推理；
5. 看不清核心物证时，通过动作链和后续反应辅助判定性质。

这样做的目标，是让模型学到“如何判断”，而不是记住“这道题怎么答”。

### 15.5 如何判断一类边界是不是参数量主导

建议在实验设计里加入“边界来源判断”环节，而不是默认所有低分点都可通过训练修复。

可采用以下判别方法。

#### 1. 跨模型规模对比

对同一类样本，比较不同参数规模模型的表现：

1. 7B
2. 14B
3. 27B
4. 更大模型

如果表现随规模扩大而显著单调改善，说明该类任务高度 scale-sensitive。

#### 2. 同模型、不同推理框架对比

对同一模型，测试：

1. 直接回答
2. 拆步回答
3. 先子判断定位，再给结论
4. 先候选缺口比较，再回答

如果仅靠推理框架就能显著改善，说明这类边界不完全是参数硬上限，而存在明显的系统优化空间。

#### 3. 小规模针对性训练后的 held-out 泛化

不要只看训练样本本身提没提升，而要看同类未见变体是否提升：

1. 若只在原题附近改善，说明是在补题；
2. 若在同类新变体上也稳定提升，说明是在补能力。

#### 4. 错误模式是否稳定同构

若不同题上反复出现同一种错法，例如：

1. 总把可疑线索写成可下结论事实；
2. 总选不准最关键缺口；
3. 总被显眼动作带偏而漏掉真正主轴；

这通常意味着它更可能是可训练边界，而不是完全由参数量决定的“随机失败”。

### 15.6 面对不同类型边界的优化策略

#### 对可训练边界

优先策略：

1. 扩充同类能力族数据；
2. 让训练样本覆盖多个变体，而不是单题重复；
3. 强化“错误层级 -> 正确层级”的对比监督；
4. 加入结构化判断链样本。

#### 对规模敏感边界

优先策略：

1. 降低对单次自由生成的依赖；
2. 引入推理拆解；
3. 评估是否需要更大模型；
4. 不要对纯数据微调抱过高预期。

#### 对系统级边界

优先策略：

1. 设计更强的中间判别步骤；
2. 采用路由机制，把高风险题交给更强模型；
3. 增加外部验证模块或结构化中间层；
4. 优化 prompt，不让模型一次承担过多混合任务。

### 15.7 这对项目意味着什么

因此，本项目不应把“找到 Qwen3.6-27B 的能力边界”理解为：

1. 找到几道难题；
2. 然后回灌训练；
3. 期待模型自然泛化。

更合理的目标应当是：

1. 找到稳定、可归因的边界样本；
2. 抽象出边界背后的失败机制；
3. 将这些机制归类为可训练、规模敏感或系统级；
4. 根据边界类型选择不同优化路线。

换句话说，本项目的价值不只是“证明小模型有边界”，而是帮助团队明确：

1. 哪些边界值得做数据训练；
2. 哪些边界主要受模型规模限制；
3. 哪些边界更适合通过系统设计解决；
4. 哪些方向继续投入数据和算力是高收益的，哪些方向则应及时止损。

### 15.8 建议新增的实验输出

为了让上述框架真正落地，建议在每轮分析中为边界样本新增以下字段：

1. `boundary_type`
   - trainable
   - scale_sensitive
   - system_level
   - undecided

2. `optimization_recommendation`
   - data
   - larger_model
   - reasoning_decomposition
   - routing
   - mixed

3. `generalization_expectation`
   - high
   - medium
   - low

4. `evidence_for_boundary_type`
   - 用简短文字记录为什么判断它属于该类边界

这些字段将帮助项目从“题目进化流水线”逐步升级成“能力边界分析与优化决策流水线”。

## 16. 实施优先级建议

建议按以下顺序推进。

### 第一优先级

1. 建立原始样本准入筛选规则；
2. 为样本补充结构化画像；
3. 明确“哪些题不值得进化”。

### 第二优先级

1. 把进化策略拆成 operator taxonomy；
2. 建立基础路由规则；
3. 避免单 prompt 通吃所有样本。

### 第三优先级

1. 增加复杂度预算硬校验；
2. 增加停止条件；
3. 把不可回答和无效复杂化尽早拦住。

### 第四优先级

1. 建立经验库；
2. 做算子效果矩阵；
3. 将实验分析结果反哺下一轮路由与 prompt 更新。

### 第五优先级

1. 做 judge 校准；
2. 做 rubric 稳定性分析；
3. 提升整体实验结论的可信度。

## 17. 推荐的近期落地动作

如果只做最小可行改造，建议先完成以下四项：

1. 给现有子集样本补一层 `sample_profile` 字段；
2. 把当前 evolution prompt 中的算子显式化，至少区分 5-7 类主方式；
3. 在代码层增加 evolved question 的预算校验；
4. 新增一份结构化实验结果表，记录“样本类型 x 算子 x 效果”。

## 18. 总结

这个项目下一阶段最重要的，不是继续追求“题目更难”，而是把以下闭环做稳：

1. 先筛掉不值得进化的题；
2. 给值得进化的题做清晰画像；
3. 基于画像选择合适算子；
4. 用复杂度预算约束进化；
5. 用可归因的评分验证是否真的打到能力边界；
6. 把成功和失败都沉淀为经验资产。

最终目标应当是：

1. 样本不是越来越多，而是越来越值钱；
2. 题目不是越来越长，而是越来越尖；
3. 分数不是单纯更低，而是更能说明模型到底不会什么。
