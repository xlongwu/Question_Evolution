# 阶段06：测试验证、迁移兼容与实验验收

阶段编号：06

阶段名称：测试验证、迁移兼容与实验验收

阶段目标：用系统化验证确认树状搜索改造没有破坏原有单链流程，并能稳定产出一个样本多个能力边界点的可复盘结果。

阶段范围：覆盖单元测试、集成样例、迁移说明、实验验收报告和风险收尾，不新增核心功能。该阶段也负责固化前面阶段积累的 fixture 和验收命令。

主要任务：

1. 补充 JSONL schema 与状态更新测试，验证旧单链数据和新树状搜索数据都能解析。
2. 为画像路由、候选生成、边界识别、frontier 更新建立 mock 测试。
3. 准备 tiny sample 端到端验证，控制样本数量、分支数、深度和候选预算。
4. 验证旧输入数据兼容，确认禁用树状搜索时旧流程可以运行。
5. 验证启用树状搜索时，同一样本能够产生多分支、多能力轴边界或正确停止。
6. 检查成本预算、停止条件和重复边界控制，避免小样本以外的成本失控。
7. 编写迁移兼容说明、默认参数建议和最终验收报告。

涉及模块/文件范围：`tests/`、小型 fixture、阶段01到阶段05涉及脚本、`run_loop.sh`、实验输出目录。

前置依赖：依赖阶段01到阶段05主要功能完成。也可以在每个阶段内同步积累对应测试，最后集中补齐端到端验收。

完成标准：

1. 核心结构逻辑有 focused pytest 或等价测试覆盖。
2. 禁用树状搜索时旧流程可运行。
3. 启用树状搜索时至少能观察到同一样本多分支、多边界或正确停止。
4. `search_graph.jsonl` 能复盘父子关系、边界轴、选中状态和停止原因。
5. 验收文档说明已知风险、默认参数、运行方式和后续调参建议。

测试与验证方式：

1. 运行 focused pytest，覆盖 schema、动作决策、候选 metadata、边界去重和 frontier 更新。
2. 用 mock LLM/API 测试结构逻辑，避免真实模型输出不稳定影响代码正确性判断。
3. 用极小样本执行一次完整流程，检查输出文件是否齐全。
4. 对比树状搜索关闭和开启两种配置下的行为差异。
5. 检查 `search_graph.jsonl` 和 `active_frontier.jsonl` 的行数、节点关系和状态字段。

风险与注意事项：

1. 真实 LLM 输出不稳定，测试应优先 mock 结构逻辑，再做少量真实流程验证。
2. 端到端测试要控制样本数量和预算，不要用大规模实验作为代码正确性的唯一验证。
3. 如果前面阶段没有同步补测试，本阶段可能暴露较多返工点，需要预留修复时间。
4. 验收应同时关注“能发现多个边界”和“能及时停止”，不能只追求分支数量。

预计交付物：

1. 测试用例和 fixture。
2. 小样本端到端结果。
3. 迁移兼容说明。
4. 最终验收报告。
5. 默认参数和调参建议。

Codex Goal 建议：适合作为一次独立 Codex Goal 执行。该阶段是收尾验收，不建议与阶段05合并，否则集成问题和测试问题会混在一起，降低复盘效率。
