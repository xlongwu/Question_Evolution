# 阶段01：基础结构与搜索状态契约

阶段编号：01

阶段名称：基础结构与搜索状态契约

阶段目标：先建立本次树状搜索改造的统一数据契约，避免后续脚本各自发明字段。该阶段需要确认现有代码中真实的记录结构、状态字段、配置入口和 JSONL 读写模式，再落定 `evolution_state` 扩展、`search_graph.jsonl`、`active_frontier.jsonl` 与预算参数。

阶段范围：只处理基础契约、兼容策略、配置与最小样例，不实现具体候选生成、评分分析或复杂调度。阶段结果应能作为后续画像、路由、生成、选择、状态更新的共同接口。

主要任务：

1. 梳理现有脚本的输入输出文件、核心 record 字段和 `evolution_state` 当前结构。
2. 定义树状搜索核心字段，包括 `search_root_id`、`current_node_id`、`parent_node_id`、`branch_id`、`boundary_axis`、`branch_status`、`search_depth`、`max_search_depth`、`branch_budget_remaining`、`sample_budget_remaining`、`discovered_boundaries`、`recommended_next_axes`、`stop_status`。
3. 定义 `search_graph.jsonl` 节点行格式，覆盖 `sample_id`、`node_id`、`parent_node_id`、`prompt`、`operator_used`、`effect_label`、`is_boundary_hit`、`selected_into_mainline`、`dedup_signature` 等字段。
4. 定义 `active_frontier.jsonl` 待扩展节点格式，明确 source node、目标能力轴、动作类型、预算快照和所需 prompt 来源。
5. 规划新增配置项：`MAX_SAMPLE_BRANCHES`、`MAX_SAMPLE_DEPTH`、`MAX_SAMPLE_BOUNDARIES`、`MAX_SAMPLE_CANDIDATES_TOTAL`、`MAX_NO_NEW_BOUNDARY_ROUNDS`、`MAX_GLOBAL_NEW_BOUNDARY_GAP`、`ENABLE_BRANCH_BACKTRACK`、`ENABLE_ROOT_FORK`。
6. 设计旧单链数据兼容策略，对缺失树状搜索字段的记录提供默认 root、默认 branch 和默认当前节点。
7. 准备 2 到 3 条最小 JSONL 样例，用于表示根节点、子分支、已发现边界和待扩展 frontier。

涉及模块/文件范围：`question_evolution.py`、`run_loop.sh`、`profile_samples.py`、`select_evolution_candidates.py`、`operator_router.py`、`candidate_selection.py`、`analyze_evolution_effect.py`、`update_sample_state.py`、可能存在的配置文件、`data/` 小样例和 `experiments/` 输出约定。

前置依赖：需要先确认当前仓库内上述脚本是否全部存在，以及字段命名是否与方案文档一致。若某些脚本尚不存在，应在本阶段明确替代模块或新增模块的位置。

完成标准：

1. 形成后续阶段共同遵循的状态字段契约。
2. `search_graph.jsonl` 和 `active_frontier.jsonl` 的字段、含义、必填性和默认值明确。
3. 旧单链 record 在缺少新字段时可以被安全解析和补齐。
4. 新配置项有默认值、含义说明和关闭树状搜索的回退路径。
5. 最小样例能够表达根节点、分支节点、边界命中、重复分支和待扩展节点。

测试与验证方式：

1. 用手工样例验证 schema 的完整性和可读性。
2. 对旧单链 record 做只读解析验证，确认不会因为缺失新字段报错。
3. 检查新增字段不会破坏答案收集、rubric 生成和 scoring 阶段需要的原字段。
4. 用样例手动追踪 `node_id`、`parent_node_id`、`branch_id` 是否能复盘一棵浅树。

风险与注意事项：

1. 字段契约一旦落地到多个脚本，后续返工成本很高，因此本阶段不宜跳过。
2. `node_id` 和 `branch_id` 必须可追踪，避免并发或重跑时产生冲突。
3. `search_graph.jsonl` 应作为搜索轨迹文件，不应取代主链输出，否则会扩大迁移范围。
4. 预算字段需要区分单分支预算和单样本总预算，避免后续重复扣减。
5. 需要人工确认树状搜索默认关闭还是默认开启。

预计交付物：

1. 搜索状态字段说明。
2. `search_graph.jsonl` 格式说明。
3. `active_frontier.jsonl` 格式说明。
4. 预算参数说明表。
5. 旧数据兼容策略。
6. 最小 JSONL 样例。

Codex Goal 建议：适合作为一次独立 Codex Goal 执行。该阶段是后续所有阶段的基础，不建议再拆成多个微任务，除非现有代码结构与方案差异很大。
