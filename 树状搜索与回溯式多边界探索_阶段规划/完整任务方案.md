# 树状搜索与回溯式多边界探索改造方案

## 1. 方案目标

当前项目已经具备以下能力：

1. 能对单个样本进行多轮 question evolution；
2. 能在某一轮内对单样本并行生成多个候选题；
3. 能识别一部分有效能力边界样本；
4. 能在进化失败后换算子重试；
5. 能在命中明显边界后停止继续进化。

但当前机制仍然以“单链主路径”为主。一个样本在某一条路径上打到一个边界后，往往就停止继续探索，或者后续只做透传与轻微换算子。这会导致两个问题：

1. 一个原始样本通常只能产出一个主要边界点；
2. 一旦当前链路命中了某个边界，系统很难再回到原始题，继续寻找这个样本可能隐含的其他能力边界。

本方案的目标是把当前项目从“单链式 question evolution”升级为“链式主流程 + 局部树状搜索 + 回溯式多边界探索”的混合机制，使得：

1. 同一个原始样本可以挖出多个能力边界点；
2. 命中一个边界后，不代表该样本整体结束，而是可以回到根节点或较早节点继续开新分支；
3. 项目不再主要依赖全局 round 数增加，而是主要依赖单样本搜索预算；
4. 最终产物不只是“一条进化链”，而是“一个样本对应的多条有效边界分支集合”。

---

## 2. 为什么现在要改

结合当前实验结果，单链式机制已经暴露出明显上限。

### 2.1 当前机制的优点

1. 易于实现；
2. 单条轨迹清晰；
3. 每轮状态更新简单；
4. 成本相对可控。

### 2.2 当前机制的主要问题

1. **命中一个边界后，样本往往被整体停掉**
   例如像 `18318`、`859` 这类样本，一旦当前链路已经暴露出 Qwen 的某个失误点，后面通常进入停止或透传状态。这样虽然保住了已有成果，但也放弃了同一样本里可能还存在的其他边界。

2. **换算子仍然主要沿着当前链路继续走**
   即使状态里写了 `recommended_next_methods`，它本质上还是“在当前题的后继节点上继续改”，不是“回到更早节点开一条新分支”。

3. **全局 round 机制容易浪费轮次**
   如果继续提高 `MAX_ROUNDS`，大量轮次会被花在已经没有信息增量的样本上，而不是真正服务于边界探索。

4. **一个样本难以沉淀成多能力边界样本池**
   现在更像“一个样本命中一个主要边界”，而不是“一个样本可对应多个边界方向：分层错误、关键缺口错误、伪闭环错误、补强升级判断错误”等。

### 2.3 为什么不能只靠把 round 数调高解决

如果只提高全局轮数，而不改变搜索结构，会出现：

1. 已停样本继续透传；
2. 未停样本继续在同一链路上换壳追问；
3. 信息增量越来越低；
4. 计算成本越来越高；
5. 难以判断“这个样本到底是已探明边界，还是只是当前链条走到头了”。

所以，后续不应主要增加全局 round，而应增加“单样本可控的局部搜索预算”。

---

## 3. 改造后的总体机制

改造后的项目总体上仍然保留链式主流程，但要把“单样本探索单元”从一条链改成一棵浅树。

### 3.1 新的总体思路

对每个原始样本：

1. 保留一个根节点：原始题；
2. 从根节点出发，针对不同能力方向生成 2 到 4 条一级分支；
3. 每条分支继续小深度扩展；
4. 某条分支命中一个有效边界后，记录为一个“已发现边界点”；
5. 该边界点不会阻止系统回到根节点或更早节点，继续探索其他尚未命中的方向；
6. 当该样本的搜索预算耗尽，或已无新边界可挖时，才停止整个样本。

### 3.2 新机制不是纯树搜索

这里不建议做“深而宽”的通用树搜索，而是做“浅层、受预算约束的局部树状探索”。

原因是：

1. 当前项目需要的是可复盘、可解释、可稳定评分的边界样本，不是无限探索；
2. 真正有价值的是“一个样本挖出 2 到 4 个不同边界点”，而不是对一个样本展开十几层树；
3. 多数题目没有必要做深树，浅树就足够。

因此推荐结构是：

1. 单样本最大深度：2 到 3；
2. 单节点最大分支数：2 到 4；
3. 单样本最大有效边界点数：2 到 4；
4. 单样本最大总候选数：例如 8 到 12。

## 4. 新的数据结构设计

当前项目的状态字段主要面向单链，需要扩成“单样本搜索图”。

### 4.1 新增核心概念

对每个样本新增以下概念：

1. `search_root_id`
   表示这个样本搜索树的根节点，对应原始题。

2. `node_id`
   表示当前题目节点的唯一 ID。

3. `parent_node_id`
   表示该节点由哪个上游节点进化而来。

4. `branch_id`
   表示该节点属于哪一条分支。

5. `boundary_axis`
   表示该分支主要在压哪一种能力边界，例如：
   - 结论分层
   - 最关键缺口识别
   - 伪闭环识别
   - 补强项升级判断
   - 题干外补设识别
   - 反常线索主线切换

6. `branch_status`
   表示这条分支当前状态，例如：
   - exploring
   - boundary_hit
   - exhausted
   - duplicate
   - invalid

7. `search_budget`
   表示当前样本剩余预算。

8. `discovered_boundaries`
   表示当前样本已发现的边界点集合。

### 4.2 建议新增的状态字段

建议在 `evolution_state` 基础上扩成如下结构：

```json
{
  "search_root_id": "sample_18318_root",
  "current_node_id": "sample_18318_b2_d1",
  "parent_node_id": "sample_18318_root",
  "branch_id": "branch_subclaim_01",
  "boundary_axis": "结论分层",
  "branch_status": "exploring",
  "search_depth": 1,
  "max_search_depth": 3,
  "branch_budget_remaining": 2,
  "sample_budget_remaining": 8,
  "discovered_boundaries": [
    {
      "boundary_id": "b_001",
      "boundary_axis": "结论分层",
      "trigger_node_id": "sample_18318_b1_d1",
      "effect_label": "effective_boundary_probe"
    }
  ],
  "recommended_next_methods": ["O4_near_level_ranking"],
  "recommended_next_axes": ["关键缺口识别", "相邻层级细分"],
  "stop_status": "continue_branch_search"
}
```

### 4.3 单独维护搜索图文件

除了在单条 record 里放状态，建议新增一个独立文件：

`search_graph.jsonl`

每一行记录一个节点，字段包括：

1. `sample_id`
2. `search_root_id`
3. `node_id`
4. `parent_node_id`
5. `branch_id`
6. `depth`
7. `prompt`
8. `operator_used`
9. `boundary_axis`
10. `score_rate_before`
11. `score_rate_after`
12. `effect_label`
13. `is_boundary_hit`
14. `selected_into_mainline`
15. `dedup_signature`

这样可以在不污染主链数据结构的前提下，保留完整搜索轨迹。

## 5. 主流程要怎么改

### 5.1 入口层：不再把“当前题”当成唯一下一步起点

当前流程默认输入是上一轮 `state_updated.jsonl`，每个样本只有一个当前 prompt。

改造后应允许：

1. 某些样本继续沿当前节点扩展；
2. 某些样本回到根节点重新开一条新分支；
3. 某些样本从中间层节点回溯，开兄弟分支。

因此，下一轮不应只消费“上一轮主链输出”，而应消费“待扩展节点池”。

建议新增：

`active_frontier.jsonl`

含义是：下一轮真正需要展开的节点集合。

### 5.2 从“全局 round”改成“frontier 扩展”

保留 round 作为实验批次概念可以，但不要再把它理解成“每个样本都统一前进一步”。

新的 round 更像：

1. 从 `active_frontier` 中取出一批待扩展节点；
2. 对这些节点做画像、分流、路由、候选生成、校验、评分；
3. 根据结果决定：
   - 某分支继续深入；
   - 某分支停止；
   - 从根节点再开新分支；
   - 整个样本停止。

## 6. 各脚本的改造点

### 6.1 `profile_samples.py`

当前功能：

1. 给样本打画像；
2. 判断 overscore 主因。

改造后新增：

1. 增加 `boundary_axis_candidates`
   为每个样本给出 2 到 4 个可能的能力边界方向。

2. 增加 `already_explored_axes`
   结合 `evolution_state.discovered_boundaries`，标出哪些方向已经试过。

3. 增加 `next_best_axes`
   推荐下一步优先探索的能力轴，而不仅仅是推荐算子。

### 6.2 `select_evolution_candidates.py`

当前功能：

1. 判断当前样本是高分虚高、低分真实边界、透传还是停止。

改造后新增：

1. 决策不再只是“当前题要不要继续 evolve”，而是：
   - 当前节点继续扩展；
   - 回根节点开新分支；
   - 回父节点开兄弟分支；
   - 停止整个样本。

2. 新增动作类型，例如：
   - `expand_current_branch`
   - `fork_from_root`
   - `fork_from_parent`
   - `stop_branch`
   - `stop_sample`

### 6.3 `operator_router.py`

当前功能：

1. 给当前题选主算子和 backup 算子；
2. 标记是否值得做局部树状探索。

改造后新增：

1. 路由输入要显式包含 `boundary_axis`，而不只是 `candidate_overscore_cause`。

2. router 输出不再只是算子，还要输出：
   - 本次是继续当前分支还是新开分支；
   - 新分支优先从根节点还是父节点出发；
   - 新分支目标能力轴。

3. 若某个能力轴已经命中过，不应再优先重复同一轴，除非明确要做更细分层。

### 6.4 `question_evolution.py`

当前功能：

1. 对一个样本生成 1 到 4 个候选题；
2. 每个候选只对应当前节点的直接后继。

改造后新增：

1. 支持从指定 source node 生成候选，而不是默认从当前主链题生成。

2. 每个候选要记录：
   - 来源节点；
   - 分支类型；
   - 目标边界轴；
   - 是否属于新分支。

3. 新增“回溯生成”模式：
   - 以 `prompt_old` 或根节点 prompt 为源题；
   - 重新按新的能力轴开题。

### 6.5 `candidate_selection.py`

当前功能：

1. 在单轮局部候选中选一个进入主链。

改造后新增：

1. 不只是选“哪个题进入主链”，还要判断：
   - 这个候选是否形成了一个新的边界类型；
   - 这个候选是否和已有边界点重复；
   - 这个候选是否值得保留为边界样本，但不继续深入。

2. 输出应同时支持：
   - `selected_into_mainline = true`
   - `selected_as_boundary_leaf = true`
   - `discard_as_duplicate = true`

### 6.6 `analyze_evolution_effect.py`

当前功能：

1. 判断有没有降分；
2. 判断是否形成 `effective_boundary_probe`。

改造后新增：

1. 不再只判断“当前链路是否有效”，还要判断：
   - 这个边界点属于哪一类能力轴；
   - 是否与本样本已发现边界重复；
   - 是否构成“新边界点”。

2. 新增两个重要输出字段：
   - `boundary_axis_detected`
   - `is_new_boundary_for_sample`

### 6.7 `update_sample_state.py`

这是改造的核心脚本之一。

新增职责：

1. 维护单样本搜索预算；
2. 决定下一步是继续当前分支、回根节点、开新分支，还是停止整个样本；
3. 维护 `discovered_boundaries`；
4. 维护 `active_frontier`；
5. 判断是否所有边界都已探明。

## 7. 停止条件怎么设计

### 7.1 单分支停止条件

某一条分支可以停止，如果满足以下任一条件：

1. 已命中有效边界，且继续沿该分支没有更细层级可压；
2. 连续 2 次扩展没有新信息；
3. 继续扩展只是在重复同家族题型；
4. 候选题开始高度同质化；
5. 已达到该分支最大深度。

### 7.2 单样本停止条件

某个原始样本整体停止，如果满足以下任一条件：

1. 已发现边界点数量达到上限；
2. 所有推荐能力轴都已探索；
3. 剩余可开新分支都只会重复已有边界；
4. 样本预算耗尽；
5. 连续若干轮没有新增边界点。

### 7.3 全实验停止条件

整个实验停止，如果满足以下任一条件：

1. 所有样本都进入 `stop_sample`；
2. 全局预算耗尽；
3. 连续若干轮没有新增有效边界点；
4. 新增边界点数低于阈值且持续稳定。

## 8. round 机制要不要保留

建议保留，但弱化其地位。

### 8.1 为什么保留

1. 方便管理批量计算；
2. 方便落地到现有脚本；
3. 方便记录实验批次。

### 8.2 为什么不能继续强依赖 round

因为树状搜索的核心不是“每个样本都同步前进一层”，而是“哪些 frontier 节点值得继续扩展”。

### 8.3 建议做法

1. 保留全局 `MAX_ROUNDS`，例如 5 到 8；
2. 但同时新增：
   - `MAX_SAMPLE_BRANCHES`
   - `MAX_SAMPLE_DEPTH`
   - `MAX_SAMPLE_BOUNDARIES`
   - `MAX_SAMPLE_CANDIDATES_TOTAL`
   - `MAX_NO_NEW_BOUNDARY_ROUNDS`

真正的停止主依据应逐步从 `MAX_ROUNDS` 转向“单样本预算 + 无新边界停止”。

---

## 9. 推荐新增的配置项

建议在配置中新增以下参数：

```bash
MAX_SAMPLE_BRANCHES=4
MAX_SAMPLE_DEPTH=3
MAX_SAMPLE_BOUNDARIES=3
MAX_SAMPLE_CANDIDATES_TOTAL=10
MAX_NO_NEW_BOUNDARY_ROUNDS=2
MAX_GLOBAL_NEW_BOUNDARY_GAP=2
ENABLE_BRANCH_BACKTRACK=true
ENABLE_ROOT_FORK=true
```

含义：

1. `MAX_SAMPLE_BRANCHES`
   单样本最多保留多少条有效分支。

2. `MAX_SAMPLE_DEPTH`
   单分支最大深度。

3. `MAX_SAMPLE_BOUNDARIES`
   单样本最多记录多少个已发现边界点。

4. `MAX_SAMPLE_CANDIDATES_TOTAL`
   单样本总候选预算。

5. `MAX_NO_NEW_BOUNDARY_ROUNDS`
   单样本连续多少轮无新边界后停止。

6. `ENABLE_BRANCH_BACKTRACK`
   是否允许从父节点回溯开兄弟分支。

7. `ENABLE_ROOT_FORK`
   是否允许从原始题重新开新能力轴分支。

---

## 10. 最终判断

后续优化方向确实应该从“继续增加 round”转向“引入树状搜索与回溯式多边界探索”。

更准确地说，项目下一阶段不该追求：

1. 把单条链跑得更长；
2. 让所有样本同步跑更多轮。

而应该追求：

1. 让高价值样本从一个根题里挖出多个不同边界点；
2. 让停止条件从“当前链路是否停”升级为“这个样本还有没有新的边界方向值得继续开分支”；
3. 让实验预算更多花在“新边界探索”上，而不是花在“重复透传和低信息增量轮次”上。

一句话概括就是：

下一阶段最合理的升级，不是把当前单链流程拉长，而是把单样本的局部探索结构从“一条链”升级成“一棵浅树”，并且允许在命中一个边界后回到根节点继续找下一个边界。
