# 阶段03：回溯式候选生成与分支元数据

阶段编号：03

阶段名称：回溯式候选生成与分支元数据

阶段目标：让问题进化不再默认只从当前主链题继续，而是可以基于 `active_frontier` 指定的 source node，从根节点、父节点或当前节点生成带分支信息的候选题。

阶段范围：覆盖候选生成入口、source prompt 选择、回溯生成模式和候选元数据写入，不负责判断候选是否命中新边界，也不负责更新下一轮 frontier。

主要任务：

1. 调整 `question_evolution.py` 的输入模型，使其可以消费 `active_frontier.jsonl` 中的待扩展节点。
2. 根据动作类型选择候选生成源题：当前节点 prompt、父节点 prompt、根节点 prompt 或旧流程中的 `prompt_old`。
3. 支持“回根节点开新分支”和“从父节点开兄弟分支”的回溯生成模式。
4. 为每个候选写入 `source_node_id`、`parent_node_id`、`branch_id`、`boundary_axis`、`is_new_branch`、`operator_used`、`search_depth`、`generation_action` 等元数据。
5. 支持同一样本不同能力轴并行生成候选，同时限制单节点分支数和单样本总候选数。
6. 保留旧单链输入兼容路径，缺少 frontier 时仍可从当前题生成候选。
7. 记录生成失败、算子重试和 fallback 算子使用情况，供后续状态更新判断分支是否 exhausted 或 invalid。

涉及模块/文件范围：`question_evolution.py`、`active_frontier.jsonl`、`search_graph.jsonl`、可能的 prompt 模板或算子配置、`run_loop.sh` 中 evolution 阶段的输入输出约定。

前置依赖：依赖阶段01字段契约和阶段02动作输出。需要先确认现有 `question_evolution.py` 是否已经支持多候选并发、算子重试、operator metadata 和 JSONL streaming。

完成标准：

1. 给定同一个样本的多个 frontier 节点，可以生成多条不同分支候选。
2. 每个候选都能追溯到 `search_root_id`、`source_node_id`、`parent_node_id` 和 `branch_id`。
3. 根节点 fork、父节点 fork、当前分支扩展三种模式都有明确行为。
4. 候选数量受单样本预算和单节点分支数约束。
5. 旧单链输入仍可按默认当前节点生成候选，不破坏现有流程。

测试与验证方式：

1. 用固定输入样例验证 `expand_current_branch`、`fork_from_root`、`fork_from_parent` 三种动作。
2. mock API 响应，验证生成内容之外的 metadata 写入完整。
3. 检查候选数量不超过 `MAX_SAMPLE_CANDIDATES_TOTAL`、`MAX_SAMPLE_BRANCHES` 和节点分支限制。
4. 验证生成失败或无效输出时不会丢失 source node 和动作信息。

风险与注意事项：

1. 回溯源 prompt 选择错误会污染整棵搜索树，需要在输出中保留 source prompt 追踪字段。
2. 并发生成时要避免 `node_id` 和 `branch_id` 冲突。
3. 同一样本多分支候选会显著增加 API 成本，必须保留预算门控。
4. 不要在本阶段判断候选是否是新边界，否则会和阶段04职责重叠。

预计交付物：

1. 支持 source node 的候选生成流程。
2. 回溯生成模式。
3. 候选分支元数据。
4. 兼容旧单链输入的默认路径。
5. 候选生成样例。

Codex Goal 建议：适合作为一次独立 Codex Goal 执行。若现有 `question_evolution.py` 很大，可以在同一 Goal 内先做结构梳理再实现，不建议拆成只改某个函数的微任务。
