# 阶段02：能力轴画像、候选分流与路由决策

阶段编号：02

阶段名称：能力轴画像、候选分流与路由决策

阶段目标：让系统从“当前题是否继续 evolve”升级为“该样本还有哪些能力轴值得探索、下一步应从当前节点还是回溯节点展开”。该阶段形成树状搜索的决策层，为候选生成阶段提供明确动作、能力轴和算子选择。

阶段范围：覆盖样本画像、候选筛选、动作类型和算子路由，不直接生成候选题，也不更新全局 frontier。阶段结果应能独立输出下一步探索建议。

主要任务：

1. 在 `profile_samples.py` 的画像结果中增加 `boundary_axis_candidates`，为样本推荐 2 到 4 个可能能力边界方向。
2. 增加 `already_explored_axes`，结合 `evolution_state.discovered_boundaries` 标出已经尝试或已经命中的能力轴。
3. 增加 `next_best_axes`，输出下一步优先探索的能力轴，而不仅是推荐算子。
4. 在 `select_evolution_candidates.py` 中把决策从“是否继续 evolve”扩展为动作类型：`expand_current_branch`、`fork_from_root`、`fork_from_parent`、`stop_branch`、`stop_sample`。
5. 在 `operator_router.py` 中显式消费 `boundary_axis`、已探索能力轴、branch 状态和预算信息。
6. 扩展 router 输出，使其包含 branch intent、source node 类型、目标能力轴、主算子、备选算子和停止原因。
7. 为 LLM 输出设计稳健解析和兜底规则，避免画像或路由输出不合法时中断流程。

涉及模块/文件范围：`profile_samples.py`、`select_evolution_candidates.py`、`operator_router.py`、阶段01定义的 `evolution_state`、`active_frontier.jsonl` 字段和预算配置。

前置依赖：依赖阶段01的数据字段、预算字段和能力轴枚举。需要人工确认能力轴分类是否以方案中的 6 类为默认集合，还是要结合已有 prompt 分类扩展。

完成标准：

1. 每个待探索样本都能得到可解释的能力轴候选。
2. 已命中的能力轴不会被默认重复探索，除非明确标记为更细层级探索。
3. 决策输出能够区分继续当前分支、回根节点开新分支、回父节点开兄弟分支和停止。
4. router 输出足以驱动阶段03选择 source prompt 和 operator。
5. 预算不足、无可探索能力轴、重复能力轴等情况有明确停止或降级动作。

测试与验证方式：

1. 使用包含未探索、已命中边界、重复能力轴、预算不足、无推荐轴等状态的样例验证动作输出。
2. 使用 mock LLM 画像或固定 JSON 响应验证解析稳定性。
3. 检查动作分布是否符合预算约束，不应在预算耗尽时继续 fork。
4. 验证 router 在 `boundary_axis` 缺失时有安全兜底，例如退回当前单链算子逻辑。

风险与注意事项：

1. 能力轴分类过细会导致搜索碎片化，过粗会导致多边界去重失效。
2. 画像和路由如果直接修改预算，会造成后续状态更新重复扣减。本阶段只输出建议，不消费预算。
3. LLM 输出可能含有不在枚举内的能力轴，需要映射、降级或保留原始审计字段。
4. `fork_from_root` 与 `fork_from_parent` 的选择会影响成本和边界多样性，需要保留可解释原因。

预计交付物：

1. 能力轴画像输出结构。
2. `already_explored_axes` 和 `next_best_axes` 计算逻辑。
3. 树状搜索动作类型定义。
4. router 扩展输出结构。
5. 样例决策结果和兜底规则。

Codex Goal 建议：适合作为一次独立 Codex Goal 执行。阶段内容属于同一个“决策层闭环”，不建议按三个脚本拆成三个独立 Goal。
