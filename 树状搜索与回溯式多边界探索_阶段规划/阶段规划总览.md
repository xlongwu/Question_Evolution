# 树状搜索与回溯式多边界探索阶段规划总览

## 总体拆解原则

本次改造的核心目标，是把现有单链式 question evolution 升级为“链式主流程 + 局部树状搜索 + 回溯式多边界探索”的可控搜索机制。阶段划分以可独立验收的功能闭环为单位，不按单个文件或函数拆分。

建议拆分为 6 个阶段：

1. 阶段01：基础结构与搜索状态契约
2. 阶段02：能力轴画像、候选分流与路由决策
3. 阶段03：回溯式候选生成与分支元数据
4. 阶段04：边界识别、候选选择与去重沉淀
5. 阶段05：状态更新、frontier 编排与主流程集成
6. 阶段06：测试验证、迁移兼容与实验验收

其中阶段01是后续所有实现的基础。阶段02和阶段03在阶段01完成后可以部分并行，但阶段03最终需要消费阶段02定义的能力轴与动作类型。阶段04依赖阶段03产生的分支候选与评分结果。阶段05依赖阶段02到阶段04的输出契约。阶段06贯穿各阶段做局部验证，并在最后完成端到端验收。

## 阶段总览

| 阶段 | 阶段名称 | 核心目标 | 主要交付物 | 是否核心功能 |
| --- | --- | --- | --- | --- |
| 01 | 基础结构与搜索状态契约 | 建立树状搜索数据契约、预算配置、图文件与 frontier 文件格式 | schema、配置项、兼容策略、最小样例 | 是 |
| 02 | 能力轴画像、候选分流与路由决策 | 让系统能决定探索哪个能力轴、从哪里开分支、采取什么动作 | 能力轴候选、动作类型、router 输出契约 | 是 |
| 03 | 回溯式候选生成与分支元数据 | 让 evolution 支持从根节点、父节点、当前节点生成带分支元数据的候选 | source node 生成、branch metadata、回溯模式 | 是 |
| 04 | 边界识别、候选选择与去重沉淀 | 判断候选是否形成新边界，是否重复，是否保留为边界叶子 | boundary 检测、去重签名、候选保留策略 | 是 |
| 05 | 状态更新、frontier 编排与主流程集成 | 维护预算、分支状态、已发现边界，并驱动下一轮 frontier | update state、active_frontier、run_loop 接入 | 是 |

## 阶段01：基础结构与搜索状态契约

阶段目标：先建立本次树状搜索改造的统一数据契约，避免后续脚本各自发明字段。该阶段需要确认现有代码中真实的记录结构、状态字段、配置入口和 JSONL 读写模式，再落定 `evolution_state` 扩展、`search_graph.jsonl`、`active_frontier.jsonl` 与预算参数。

阶段范围：只处理基础契约、兼容策略、配置与最小样例，不实现具体候选生成、评分分析或复杂调度。

主要任务：梳理现有脚本输入输出；定义 `search_root_id`、`node_id`、`parent_node_id`、`branch_id`、`boundary_axis`、`branch_status`、`search_depth`、`sample_budget_remaining`、`discovered_boundaries` 等字段；定义 `search_graph.jsonl` 节点行格式；定义 `active_frontier.jsonl` 待扩展节点格式；新增或规划预算配置项；制定旧单链数据的默认补齐规则。

涉及模块/文件范围：`question_evolution.py`、`run_loop.sh`、`profile_samples.py`、`select_evolution_candidates.py`、`operator_router.py`、`candidate_selection.py`、`analyze_evolution_effect.py`、`update_sample_state.py`、可能存在的配置文件、`data/` 小样例和 `experiments/` 输出约定。

前置依赖：需要先确认当前仓库内上述脚本是否全部存在，以及字段命名是否与方案文档一致。

完成标准：形成被后续阶段共同遵循的字段契约；旧数据在缺少新字段时有明确默认值；小型 JSONL 样例可以表达根节点、子节点、分支、已发现边界和待扩展 frontier。

测试与验证方式：用 2 到 3 条手工样例验证 schema；对旧单链 record 执行只读解析验证；确认新增字段不会破坏现有评分、答案收集和 rubric 生成阶段的必需字段。

风险与注意事项：字段一旦分散落地，后续返工成本高；`node_id` 和 `branch_id` 必须可复现或可追踪；`search_graph.jsonl` 不能替代主链输出，否则会扩大迁移范围。

预计交付物：搜索状态字段说明、`search_graph.jsonl` 格式说明、`active_frontier.jsonl` 格式说明、预算参数表、旧数据兼容策略、最小样例数据。

## 阶段02：能力轴画像、候选分流与路由决策

阶段目标：让系统从“当前题是否继续 evolve”升级为“该样本还有哪些能力轴值得探索、下一步应从当前节点还是回溯节点展开”。该阶段形成树状搜索的决策层。

阶段范围：覆盖样本画像、候选筛选、动作类型和算子路由，不直接生成候选题，也不更新全局 frontier。

主要任务：在画像结果中增加 `boundary_axis_candidates`、`already_explored_axes`、`next_best_axes`；在候选选择入口增加 `expand_current_branch`、`fork_from_root`、`fork_from_parent`、`stop_branch`、`stop_sample` 等动作；让 router 显式消费 `boundary_axis` 和已探索能力轴；输出目标 source node 类型、目标能力轴、主算子、备选算子和分支动作。

涉及模块/文件范围：`profile_samples.py`、`select_evolution_candidates.py`、`operator_router.py`、阶段01定义的状态契约和 frontier 契约。

前置依赖：依赖阶段01的数据字段、预算字段和能力轴枚举。需要人工确认能力轴分类是否以方案中 6 类为默认集合，还是要结合已有 prompt 分类扩展。

完成标准：每个待探索样本都能得到可解释的能力轴候选和下一步动作；已命中的能力轴不会被默认重复探索；router 输出包含后续候选生成所需的 source node、branch intent、operator 和 boundary axis。

测试与验证方式：用包含未探索、已命中边界、重复能力轴、预算不足等状态的样例验证动作输出；mock LLM 画像或使用固定 JSON 响应验证解析稳定性；检查动作分布是否符合预算约束。

风险与注意事项：能力轴分类过细会导致搜索碎片化，过粗会导致多边界去重失效；LLM 输出必须有兜底解析；动作层不能直接消耗预算，预算扣减应留到状态更新阶段统一处理。

预计交付物：能力轴画像输出、动作类型定义、router 输出结构、样例决策结果、异常输出兜底规则。

## 阶段03：回溯式候选生成与分支元数据

阶段目标：让问题进化不再默认只从当前主链题继续，而是可以基于 `active_frontier` 指定的 source node，从根节点、父节点或当前节点生成带分支信息的候选题。

阶段范围：覆盖候选生成入口、source prompt 选择、回溯生成模式和候选元数据写入，不负责判断候选是否命中新边界。

主要任务：调整 `question_evolution.py` 的输入模型，使其消费待扩展节点；根据动作类型选择 `prompt_old`、根节点 prompt、父节点 prompt 或当前节点 prompt；为每个候选写入 `source_node_id`、`parent_node_id`、`branch_id`、`boundary_axis`、`is_new_branch`、`operator_used`、`search_depth`；支持同一样本不同能力轴并行生成候选；保证候选总量受阶段01预算约束。

涉及模块/文件范围：`question_evolution.py`、`active_frontier.jsonl`、`search_graph.jsonl`、可能的 prompt 模板或算子配置、`run_loop.sh` 中 evolution 阶段的输入输出约定。

前置依赖：依赖阶段01字段契约和阶段02动作输出。需要先确认现有 `question_evolution.py` 是否已经支持多候选并发、算子重试和 operator metadata。

完成标准：给定同一个样本的多个 frontier 节点，可以生成多条不同分支候选；候选记录能追溯到 source node 和 root；旧的单链输入仍可按默认当前节点生成候选。

测试与验证方式：用固定输入样例验证根节点 fork、父节点 fork、当前分支扩展三种模式；mock API 响应验证 metadata 写入完整；检查候选数量不超过单样本和单节点预算。

风险与注意事项：回溯源 prompt 选择错误会污染整棵树；并发生成时要避免 `node_id` 冲突；同一样本多分支候选可能显著增加 API 成本，必须保留预算门控。

预计交付物：支持 source node 的候选生成流程、回溯生成模式、候选分支元数据、兼容旧单链输入的默认路径、生成样例。

## 阶段04：边界识别、候选选择与去重沉淀

阶段目标：让系统能够判断一个候选是否形成新的能力边界，是否与已有边界重复，以及是否应作为边界叶子保存而不是继续深入。

阶段范围：覆盖评分效果分析、边界轴检测、候选去重和保留策略，不负责维护下一轮 frontier。

主要任务：在效果分析中输出 `boundary_axis_detected` 和 `is_new_boundary_for_sample`；结合 `effect_label`、分数变化、能力轴和已有 `discovered_boundaries` 判断新边界；为候选生成 `dedup_signature`；扩展 candidate selection 输出 `selected_into_mainline`、`selected_as_boundary_leaf`、`discard_as_duplicate`；保留有效边界叶子，即使它不继续进入主线扩展。

涉及模块/文件范围：`analyze_evolution_effect.py`、`candidate_selection.py`、评分输出文件、`search_graph.jsonl`、`evolution_state.discovered_boundaries`。

前置依赖：依赖阶段01数据契约、阶段03候选 metadata，以及现有 scoring 输出字段。需要确认当前评分结果中是否稳定包含 `score_rate_before`、`score_rate_after` 或等价字段。

完成标准：同一样本可沉淀多个不同能力轴边界；重复候选能被标记并排除；边界叶子和主线候选可以并存；输出足以供阶段05更新状态和 frontier。

测试与验证方式：构造同轴重复、跨轴新边界、无效降分、无降分但高价值边界等样例；验证去重签名稳定；检查 `discovered_boundaries` 增量不会误覆盖历史边界。

风险与注意事项：边界去重过严会误杀相邻层级细分，过松会让样本池重复膨胀；`boundary_axis_detected` 与目标 `boundary_axis` 不一致时要保留审计字段；不能只用分数变化判断边界，否则会漏掉结构性错误。

预计交付物：边界轴检测输出、新边界判断规则、去重签名规则、候选选择扩展输出、边界叶子保存策略。

## 阶段05：状态更新、frontier 编排与主流程集成

阶段目标：把前面阶段形成的能力轴、候选、边界判断结果收束为可持续运行的树状搜索循环，统一维护预算、分支状态、已发现边界和下一轮待扩展节点。

阶段范围：覆盖 `update_sample_state.py`、`active_frontier.jsonl` 生成、`search_graph.jsonl` 追加、停止条件、`run_loop.sh` 接入和 round 语义弱化。

主要任务：实现单分支停止条件和单样本停止条件；根据候选选择结果更新 `discovered_boundaries`、`branch_status`、`sample_budget_remaining`、`branch_budget_remaining`、`no_new_boundary_rounds`；生成下一轮 `active_frontier.jsonl`；将命中边界的节点写入 `search_graph.jsonl`；在 `run_loop.sh` 中让 round 消费 frontier，而不是默认所有样本同步前进一步；保留 `MAX_ROUNDS`，但新增并优先使用单样本预算和无新增边界停止条件。

涉及模块/文件范围：`update_sample_state.py`、`run_loop.sh`、`active_frontier.jsonl`、`search_graph.jsonl`、各阶段中间产物、实验输出目录结构。

前置依赖：依赖阶段02动作输出、阶段03候选元数据和阶段04边界选择结果。需要人工确认输出目录命名、旧实验是否需要迁移，以及是否允许新增每轮 graph/frontier 文件。

完成标准：一个样本命中某个边界后，系统可以回到根节点或父节点继续探索其他能力轴；预算耗尽或无新边界时样本停止；每轮输出中可追踪 frontier、graph 和主链状态；旧单链流程可通过配置关闭树状搜索。

测试与验证方式：用 2 到 3 个样本跑 dry run 或 mock run，覆盖继续当前分支、根节点 fork、父节点 fork、停止分支、停止样本；检查预算扣减、frontier 数量、graph 节点关系和停止状态；验证禁用回溯配置时行为退回单链。

风险与注意事项：这是状态一致性风险最高的阶段；预算扣减必须集中，避免重复扣减；如果 frontier 和 graph 写入不具备幂等性，失败重跑会产生重复节点；主流程集成前应先用小样本验证。

预计交付物：状态更新闭环、frontier 编排、search graph 追加逻辑、run_loop 集成、停止条件实现、可回退配置。

## 阶段06：测试验证、迁移兼容与实验验收

阶段目标：用系统化验证确认树状搜索改造没有破坏原有单链流程，并能稳定产出一个样本多个能力边界点的可复盘结果。

阶段范围：覆盖单元测试、集成样例、迁移说明、实验验收报告和风险收尾，不新增核心功能。

主要任务：补充 JSONL schema 与状态更新测试；为画像路由、候选生成、边界识别、frontier 更新建立 mock 测试；准备 tiny sample 端到端验证；验证旧输入数据兼容；检查成本预算、停止条件和重复边界控制；编写阶段验收报告和后续调参建议。

涉及模块/文件范围：`tests/`、小型 fixture、阶段01到阶段05涉及脚本、`run_loop.sh`、实验输出目录。

前置依赖：依赖阶段01到阶段05主要功能完成。也可以在每个阶段内同步积累对应测试，最后集中补齐端到端验收。

完成标准：核心路径均有小样例验证；禁用树状搜索时旧流程可运行；启用树状搜索时至少能观察到同一样本多分支、多边界或正确停止；验收文档说明已知风险、默认参数和推荐运行方式。

测试与验证方式：运行 focused pytest；用 mock LLM/API 测试结构逻辑；用极小样本执行一次完整流程；检查 `search_graph.jsonl` 是否能复盘父子关系、边界轴和选中状态。

风险与注意事项：真实 LLM 输出不稳定，测试应优先 mock 结构逻辑；端到端测试要控制样本数量和预算；不要用大规模实验作为代码正确性的唯一验证。

预计交付物：测试用例、fixture、小样本端到端结果、迁移兼容说明、最终验收报告、默认参数建议。

## 整体执行建议

推荐实施顺序：阶段01先行；阶段02和阶段03可在阶段01完成后部分并行；阶段04在阶段03产出候选结构后推进；阶段05在阶段02到阶段04的输出稳定后集成；阶段06作为最终验收，但测试 fixture 应从阶段01开始逐步积累。

适合 Codex 直接执行的阶段：阶段01、阶段02、阶段03、阶段04、阶段05和阶段06都适合由 Codex 执行，但阶段05执行前最好先确认输出目录和兼容策略。

执行前需要人工确认的事项：能力轴枚举是否固定为方案中的 6 类；默认预算参数是否采用 `MAX_SAMPLE_BRANCHES=4`、`MAX_SAMPLE_DEPTH=3`、`MAX_SAMPLE_BOUNDARIES=3`、`MAX_SAMPLE_CANDIDATES_TOTAL=10`；旧实验输出是否需要迁移；树状搜索默认开启还是通过配置开关开启；是否允许在每轮输出目录新增 `search_graph.jsonl` 和 `active_frontier.jsonl`。

Codex Goal 拆分建议：每个阶段都适合作为一次独立 Codex Goal。阶段05如果现有 `run_loop.sh` 和 `update_sample_state.py` 较复杂，可以拆为两个执行任务：先实现状态更新和 frontier，再接入 run_loop。阶段06适合作为独立收尾 Goal。

避免阶段过碎的建议：不要按文件拆 Goal，而按“数据契约”“决策层”“生成层”“识别选择层”“状态编排层”“验收层”拆；每次 Goal 开始前只读取本阶段相关脚本和上一阶段交付文档；每次只维护一个主要输入输出契约，避免在多个阶段反复调整同一字段。

## 建议优先级

优先执行阶段01。阶段01确认字段和兼容规则后，后续阶段才不会因为 `node_id`、`branch_id`、`frontier`、`discovered_boundaries` 的定义不一致而返工。
