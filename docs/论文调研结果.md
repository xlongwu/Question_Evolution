## 论文 1. WizardLM: Empowering Large Language Models to Follow Complex Instructions

### 1. 基本信息

论文标题：WizardLM: Empowering Large Language Models to Follow Complex Instructions

作者：Can Xu, Qingfeng Sun, Kai Zheng, Xiubo Geng, Pu Zhao, Jiazhan Feng, Chongyang Tao, Qingwei Lin, Daxin Jiang

年份：2023；ICLR 2024

链接：https://arxiv.org/abs/2304.12244

来源：arXiv / ICLR 2024

一句话概括：提出 Evol-Instruct，用 LLM 将种子 instruction 逐步改写成更复杂 instruction，再用于指令微调。

### 2. 这篇论文和本项目的相关性

它和 `question_evolution_pipeline` 最相关的点是什么：最直接对应本项目的“原题 → evolved question”，尤其适合参考进化算子设计。

它更接近下面哪一类问题：指令进化、问题进化、合成数据构建、难度控制

为什么它值得调研：最直接对应本项目的“原题 → evolved question”，尤其适合参考进化算子设计。

为什么它和本项目不完全一样：它服务通用 instruction tuning，本项目服务犯罪分析任务中的 Qwen 能力边界挖掘。

### 3. 论文要解决的核心问题

这篇论文想解决什么问题：人工构造高复杂度 instruction 成本高、覆盖有限，且人类不一定能稳定写出高复杂度任务。

原有方法的痛点是什么：原有方法缺少自动化、适配性、反馈复用或稳定的样本筛选机制。

作者认为“难点”主要来自哪里：难点来自如何把模型行为反馈转化为可执行的生成、筛选、路由或训练机制。

作者希望提升的目标是什么：提升任务难度、模型能力、评测区分度、样本质量或推理稳定性。

### 4. 进化对象是什么

这篇论文进化的对象是什么：instruction；在本项目中可映射为 question。

原始输入是什么：种子 instruction。

进化后的输出是什么：更复杂、更多样的 evolved instruction。

进化前后，什么保持不变：任务目标、评测目标或核心语义应保持不变。

进化前后，什么被允许变化：中间路径、prompt、候选样本、复杂度、数据分布、反馈策略或推理方式可以变化。

### 5. 进化机制是什么

这篇论文采用什么机制：人工设计 Evol-Instruct prompt，按深度进化和广度进化改写任务，再生成回答并用于训练。

具体流程用几步说清：

1. 准备原始样本、任务、prompt 或候选池。
2. 生成候选、路径、prompt、样本或反馈。
3. 使用指标、模型、人工或环境反馈筛选候选。
4. 将保留结果用于下一轮生成、训练、选择或记忆沉淀。

如果有多轮迭代，停止条件是什么：通常由预算、轮次、性能收敛或候选质量控制。本项目应转化为 stop_status 和 max_candidate_budget。

如果有多个候选分支，如何筛选：使用任务结果、候选评估、质量过滤、一致性、难度估计或 verifier。本项目应转化为 validate_evolved_question.py 与 candidate_selection.py。

### 6. 进化方向有哪些

作者主要通过哪些方式让任务更难或更有区分度：增加约束、深化推理、增加任务组合、扩展输入复杂度、提高输出要求。

这些方向里，作者最强调的是哪一个：增加约束

这些方向哪些适合本项目：可转化为样本画像、算子路由、局部树状探索、复杂度校验或经验库。

哪些不适合本项目：不适合直接引入会修改 rubric/judge、训练模型或无限增加任务长度的做法。

原因：当前项目只优化 question evolution 生成策略，不改 rubric、judge 和评分规则。

### 7. 样本是怎么筛的

作者如何决定哪些样本值得继续进化：主要从种子 instruction 出发批量生成，样本筛选不是核心。

是否有难度估计：如果论文涉及难度、边界、不确定性或预算分配，则具有参考价值；否则需要本项目自行实现样本画像和分流。

是否有“太简单 / 太难 / 太脏”的样本剔除：HS-STaR、AutoBencher、Active Prompting 最有参考价值。

如果论文没有明确写，推测它默认依赖什么：通常依赖 benchmark 指标、人工设计数据源、模型反馈或后续过滤。

### 9. 它如何评价“进化成功”

作者用什么指标衡量成功：用训练后模型表现、人评和 GPT-4 自动评估衡量。

### 11. 它如何复用经验

有没有显式经验库、记忆库、历史轨迹保留：没有显式经验库，进化策略主要由人设计。

## 论文 15. Large Language Models have Intrinsic Self-Correction Ability

### 1. 基本信息

作者：Dancheng Liu, Amir Nassereldine, Ziming Yang, Chenhui Xu, Yuting Hu, Jiajie Li, Utkarsh Kumar, Changjae Lee, Ruiyang Qin, Yiyu Shi, Jinjun Xiong

年份：2024

链接：https://arxiv.org/abs/2406.15673

来源：arXiv

一句话概括：讨论 LLM 是否具备无需外部知识的自我修正能力，并指出零温度和公平提示影响自修正效果。

### 2. 这篇论文和本项目的相关性

为什么它和本项目不完全一样：它不是 question evolution 论文，当前阶段不引入自修正 judge。

### 3. 论文要解决的核心问题

这篇论文想解决什么问题：LLM 会幻觉，且自我修正能力是否真实存在存在争议。

原有方法的痛点是什么：原有方法缺少自动化、适配性、反馈复用或稳定的样本筛选机制。

作者认为“难点”主要来自哪里：难点来自如何把模型行为反馈转化为可执行的生成、筛选、路由或训练机制。

作者希望提升的目标是什么：提升任务难度、模型能力、评测区分度、样本质量或推理稳定性。

### 4. 进化对象是什么

这篇论文进化的对象是什么：answer / self-correction process。

原始输入是什么：模型初始回答。

进化后的输出是什么：修正后的回答。

进化前后，什么保持不变：任务目标、评测目标或核心语义应保持不变。

进化前后，什么被允许变化：中间路径、prompt、候选样本、复杂度、数据分布、反馈策略或推理方式可以变化。

### 5. 进化机制是什么

这篇论文采用什么机制：让模型在特定设置下审视并修改自身回答。

具体流程用几步说清：

1. 准备原始样本、任务、prompt 或候选池。
2. 生成候选、路径、prompt、样本或反馈。
3. 使用指标、模型、人工或环境反馈筛选候选。
4. 将保留结果用于下一轮生成、训练、选择或记忆沉淀。

如果有多轮迭代，停止条件是什么：通常由预算、轮次、性能收敛或候选质量控制。本项目应转化为 stop_status 和 max_candidate_budget。

如果有多个候选分支，如何筛选：使用任务结果、候选评估、质量过滤、一致性、难度估计或 verifier。本项目应转化为 validate_evolved_question.py 与 candidate_selection.py。

### 6. 进化方向有哪些

作者主要通过哪些方式让任务更难或更有区分度：自我反馈、自我修正、零温度、公平提示。

这些方向里，作者最强调的是哪一个：自我反馈

这些方向哪些适合本项目：可转化为样本画像、算子路由、局部树状探索、复杂度校验或经验库。

哪些不适合本项目：不适合直接引入会修改 rubric/judge、训练模型或无限增加任务长度的做法。

原因：当前项目只优化 question evolution 生成策略，不改 rubric、judge 和评分规则。

### 7. 样本是怎么筛的

作者如何决定哪些样本值得继续进化：不筛进化样本。

是否有难度估计：如果论文涉及难度、边界、不确定性或预算分配，则具有参考价值；否则需要本项目自行实现样本画像和分流。

是否有质量过滤：多数论文至少提供生成后评估或过滤思想，本项目应落实为 validation_result。

是否有去重机制：Self-Instruct 类工作更明确；其他论文需本项目自行做重复题型检测。

是否有“太简单 / 太难 / 太脏”的样本剔除：HS-STaR、AutoBencher、Active Prompting 最有参考价值。

如果论文没有明确写，推测它默认依赖什么：通常依赖 benchmark 指标、人工设计数据源、模型反馈或后续过滤。

### 8. 它如何避免无效复杂化

作者有没有意识到“更长不等于更难”：不处理题目复杂度。

有没有复杂度预算：多数论文没有本项目所需的代码级预算，因此本项目必须保留题长、主轴数、候选项数、反事实数、格式复杂度等硬校验。

1. 长度：本项目需要独立限制。
2. 任务数：本项目需要独立限制。
3. 候选项数：本项目需要独立限制。
4. 反事实数：本项目需要独立限制。
5. 新增条件数：本项目需要独立限制。
6. 输出格式复杂度：本项目需要独立限制。

如果没有，是否会对本项目构成风险：会。没有复杂度预算会导致题目变长、变脏、变格式化，最终只是在压格式而不是压能力边界。

### 9. 它如何评价“进化成功”

作者用什么指标衡量成功：看自我修正后性能是否改善。

这些指标里，哪些最适合本项目：与边界命中、样本质量、候选选择和预算分配相关的指标最适合。

哪些不够：单纯准确率、平均降分或训练后整体表现不够。

为什么：本项目要判断题目是否命中某个具体能力弱点，而不是只看平均分变化。

### 10. 它如何处理 judge / verifier / rubric 问题

论文里是否有独立评估器：涉及自评和修正，但不是 rubric 稳定性方案。

评估器是谁：可能是 benchmark、人评、LLM 自动评估、reward model、verifier 或环境反馈。

有没有讨论评分偏差：多数论文没有系统讨论，V-STaR 等 verifier 工作更接近后续阶段。

有没有讨论 verifier / judge 是否稳定：只有 verifier、自一致性、自修正相关论文更接近。

### 11. 它如何复用经验

有没有显式经验库、记忆库、历史轨迹保留：无长期 memory。

有没有把上一轮失败反馈写回下一轮：Reflexion、Auto Evol-Instruct、Promptbreeder 等工作最有参考价值。

有没有根据历史效果选择不同 operator：Automatic Instruction Evolving、Automatic Model Selection 和 Reflexion 可启发该设计。

有没有自适应调整进化策略：部分论文有，但本项目应采用规则路由 + memory bank 经验复用的受控方式。

### 12. 它是否区分“训练可修复边界”和“规模边界”

论文有没有讨论：系统级边界：自修正可能修复部分错误。

换提示/换推理流程是否能改善：Zero-shot-CoT、Self-Consistency、ToT、自修正类工作说明可以改善部分失败。

# 最终落地建议

1. 当前阶段最应直接吸收：WizardLM 的进化算子思想、Auto Evol-Instruct 的策略选择思想、Self-Instruct 的生成后过滤思想、AutoBencher 的目标化 benchmark 构建思想、Tree of Thoughts 的局部树状探索思想、Reflexion 的语言经验库思想、HS-STaR 的边界样本和预算分配思想。

2. 当前阶段暂不吸收：V-STaR 的 verifier 训练、judge agreement、rubric drift、actual failure 与 rubric item 对齐、模型训练或 SFT/RL 相关闭环。

3. 对当前方案的直接修改建议：`operator_router.py` 应读取成功和失败 memory bank；`validate_evolved_question.py` 必须执行复杂度硬校验；`candidate_selection.py` 必须在多候选中只选择一个进入主链；`analyze_evolution_effect.py` 应输出 `lightweight_boundary_hit`、`hit_confidence` 和 `needs_manual_review`；Sample Type × Operator 效果矩阵应成为每轮固定输出；三类 memory bank 必须作为数据资产沉淀，而不是临时日志。
