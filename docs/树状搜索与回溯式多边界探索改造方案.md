# 树状搜索与回溯式多边界探索改造方案

## 1. 背景与目标

当前项目已经具备完整的 question evolution 闭环：

```text
baseline scoring
-> sample profile
-> evolution action selection
-> operator routing
-> multi-candidate question evolution
-> validation
-> candidate selection
-> answer collection
-> rubric generation
-> scoring
-> effect analysis
-> state update and memory bank
```

当前实现已经支持单轮局部多候选探索。也就是说，系统可以在某一轮中为同一个样本生成 1 到 4 个候选题，再由 `candidate_selection.py` 选择一个进入主链。

但当前机制仍然以单链为主：

```text
sample -> current prompt -> candidates -> select one -> next round current prompt
```

这种机制的问题是：一旦某条链命中一个有效边界，样本通常会停止继续探索，或者只沿当前链继续换算子。它不容易回到 root prompt 或 parent prompt，继续挖同一样本里的其他能力边界。

本方案的目标是在现有功能上增量添加树搜索和回溯功能，把当前机制升级为：

```text
sample root
  -> branch A -> node A1 -> boundary hit
  -> branch B -> node B1 -> continue / stop / boundary hit
  -> branch C -> node C1 -> invalid / duplicate / retry
```

最终希望同一个原始样本可以沉淀出多个不同能力轴上的边界样本，而不只是得到一条主链。

## 2. 设计原则

1. 不推倒现有 11 步流水线。
2. 保持旧链式流程默认可用。
3. 先让搜索树可观测，再让 frontier 接管下一轮输入。
4. 先实现浅树，不做无限深树。
5. 每个阶段必须能用 pytest 验收。
6. 所有树搜索逻辑用显式配置开关控制。

建议新增总开关：

```bash
ENABLE_TREE_SEARCH=false
```

默认值为 `false`，保证旧流程不受影响。

## 3. 第一版确认范围

第一版树搜索不追求完整通用搜索，而是服务于一个明确目标：尽量从同一个原始样本中找到多个有效能力边界。

已确认的第一版取舍如下：

1. 核心目标：为单条样本发现多个不同有效边界，而不是单纯提升一条主链题目的最终难度。
2. 回溯优先级：命中一个有效边界后，优先回到 parent node 开兄弟分支；parent 不适合继续时，再回到 root node 开新能力轴分支。
3. 搜索预算采用保守方案：
   - `MAX_SAMPLE_BRANCHES=2-3`
   - `MAX_SAMPLE_DEPTH=2`
   - `MAX_SAMPLE_BOUNDARIES=2`
   - `MAX_SAMPLE_CANDIDATES_TOTAL=4-6`
4. 产物要求：保留现有流水线产物，同时新增：
   - `search_graph.jsonl`：机器可读搜索图。
   - `discovered_boundaries.jsonl`：每个样本最终发现的边界集合。
5. 第一版验证方式：可以不改真实 API 调用，优先用 fake pipeline 和小样本离线测试跑通结构。
6. 第一版验收重点：证明在现有功能基础上，树搜索和回溯逻辑能够稳定工作，包括 parent sibling fork、边界去重、预算停止和旧流程兼容。

## 4. 最终目标机制

### 4.1 当前机制

当前流程大致是：

```text
round_N/state_updated.jsonl
  -> round_N+1/input.jsonl
  -> profile / route / evolve / score / update
  -> round_N+1/state_updated.jsonl
```

每个样本在每轮通常只有一个当前 prompt。

### 4.2 改造后机制

树搜索模式下，下一轮不再只消费主链状态，而是消费 active frontier：

```text
active_frontier.jsonl
  -> profile / route / evolve / score / update
  -> search_graph.jsonl
  -> next_frontier.jsonl
```

含义是：每轮只扩展那些仍值得继续探索的节点。某个节点命中边界后，可以停止当前 branch，但不一定停止整个 sample。系统可以回到 root 或 parent 开新分支。

### 阶段 6：回溯调度器

目标：实现真正的 root/parent 回溯逻辑，生成下一轮 frontier。

新增或扩展 `frontier_scheduler.py`。

核心输入：

1. `state_updated.jsonl`
2. `search_graph.jsonl`
3. 当前 sample budget
4. discovered boundaries
5. explored axes

核心输出：

```text
next_frontier.jsonl
```

调度规则：

#### effective_boundary_probe

1. 当前 branch 标记 `boundary_hit`。
2. 如果 parent 还有可用 sibling axis，且 sample budget 未耗尽，则优先 `fork_from_parent`。
3. 如果 parent 不适合继续，再从 root fork 新分支。
4. 如果边界数量达到上限，则 `stop_sample`。

#### full_score_no_drop

1. 若深度未达上限，继续当前 branch，换 operator。
2. 若连续满分且当前 branch 无收益，回 root 开新 axis。
3. 若无新 axis，stop branch。

#### invalid_complexity

1. 当前 node 标记 invalid。
2. 若 `ENABLE_BRANCH_BACKTRACK=true`，从 parent 用 backup operator 重试。
3. 若 parent 不可用，回 root fork。

#### repeated_pattern

1. 当前 branch 标记 duplicate。
2. 回 root 换 axis。
3. 若所有 axis 都重复，stop sample。

#### no_clear_effect

1. 若 branch budget 还有，继续一次。
2. 否则 stop branch。

测试：

新增 `tests/test_tree_stage06_backtrack_scheduler.py`

覆盖：

1. boundary hit 后生成 root fork。
2. boundary hit 后优先生成 parent sibling fork。
3. invalid 后生成 parent retry。
4. repeated 后切换 axis。
5. full score 后换 operator 或 root fork。
6. budget 耗尽后不进入 next frontier。

验收标准：

### 阶段 8：fake pipeline 离线验收与真实小样本验收

#### 8.1 fake pipeline 离线验收

目标：不改真实 API 调用，证明树搜索和回溯在现有功能基础上可实现。

测试场景：

1. 一个样本 root 生成两个候选 branch。
2. branch A 命中 `effective_boundary_probe`。
3. scheduler 回到 parent 开 branch B。
4. branch B 命中另一个 axis 的 `effective_boundary_probe`。
5. `discovered_boundaries.jsonl` 中同一个 sample 有两个不同 boundary。
6. 达到 `MAX_SAMPLE_BOUNDARIES=2` 后 sample 停止。

新增测试：

```text
tests/test_tree_stage08_fake_backtrack_pipeline.py
```

验收标准：

```bash
pytest tests/test_tree_stage08_fake_backtrack_pipeline.py -q
pytest -q
```

目标：用真实 API 低并发验证树搜索模式。

验收标准：

1. `round_1/search_graph.jsonl` 存在。

6. `discovered_boundaries` 不重复记录同一 axis。
7. 预算耗尽时样本能稳定停止。

## 9. 推荐开发顺序

最稳妥的顺序是：

```text
阶段 0：树搜索开关和基线修复
阶段 1：搜索状态与 schema
阶段 2：search_graph 可观测性
阶段 3：active_frontier 兼容层
阶段 4：root fork 一层分支
阶段 5：边界去重与 boundary leaf
阶段 6：回溯调度器
阶段 7：run_loop 集成
阶段 8：fake pipeline 离线验收与真实小样本验收
```

不要一开始就把主流程改成树搜索。先让旧流程能写出搜索图，再让 frontier 接管下一轮输入，最后才启用回溯策略。

## 10. 风险与控制

### 10.1 成本风险

树搜索会增加候选数、答案采集数、rubric 生成数和评分数。必须通过以下参数限制：

1. `MAX_SAMPLE_CANDIDATES_TOTAL`
2. `MAX_SAMPLE_BRANCHES`
3. `MAX_SAMPLE_DEPTH`
4. `MAX_SAMPLE_BOUNDARIES`
5. 各阶段 concurrency

### 10.2 重复边界风险

如果没有 boundary signature，同一样本会反复挖到同类边界。必须在阶段 5 实现：

1. `boundary_axis_detected`
2. `boundary_signature`
3. `is_new_boundary_for_sample`
4. `discovered_boundaries`

### 10.3 主流程复杂化风险

不要把树搜索逻辑散落到所有脚本中。建议集中在：

1. `search_state.py`
2. `frontier_scheduler.py`
3. `build_search_graph.py`
4. `update_sample_state.py`

其他脚本只负责传递 metadata。

### 10.4 兼容性风险

所有新增字段都应该是 optional。schema 使用 `additionalProperties: true`，旧数据仍可被读取。

## 11. 最终验收标准

当以下条件满足时，可以认为树搜索和回溯功能第一版完成：

1. 旧链式流程默认仍可运行。
2. `ENABLE_TREE_SEARCH=true` 时会生成 `search_graph.jsonl`。
3. 系统能维护 `active_frontier.jsonl` 和 `next_frontier.jsonl`。
4. 同一样本可以从 root 开多个不同 branch。
5. 命中一个边界后，不会自动停止整个 sample。
6. 系统能根据 effect label 决定继续、停止、回 root、回 parent。
7. `discovered_boundaries` 能记录多个不同能力轴边界。
8. 重复边界不会被重复计入新边界。
9. 达到 sample budget 或 boundary 上限时能稳定停止。
10. fake pipeline 测试不依赖外部 API 即可覆盖 parent sibling fork、root fork 和 backtrack。
11. `discovered_boundaries.jsonl` 能按 sample 输出多个有效边界。

## 12. 一句话总结

下一阶段不要把当前单链流程简单拉长，而要把单样本探索结构从“一条链”升级成“一棵受预算约束的浅树”。第一版的核心不是追求深树，而是围绕同一样本发现多个有效边界；命中一个边界后，优先回到父节点开兄弟分支，再视情况回到 root 开新能力轴分支。
