# 树状搜索阶段06验收报告

## 验收范围

本报告对应 `树状搜索与回溯式多边界探索_阶段规划/阶段06_测试验证迁移兼容与实验验收.md`。

阶段06只做测试验证、迁移兼容说明和风险收尾，不新增核心搜索能力。核心实现边界仍归属阶段01到阶段05：

1. 阶段01定义 `evolution_state`、`search_graph.jsonl` 和 `active_frontier.jsonl` 契约。
2. 阶段02输出能力轴画像、动作决策和 router 分支提示。
3. 阶段03让 `question_evolution.py` 支持 frontier source node 生成。
4. 阶段04输出边界轴、去重签名、边界叶子和重复边界标记。
5. 阶段05集中维护预算、停止条件、graph/frontier artifacts，并接入 `run_loop.sh`。
6. 本次闭环修订把 `active_frontier` 前置为 `prepare_frontier_records.py`，使 profile/select/router/evolution 消费同一批 frontier-expanded records；候选选择移动到评分与 `effect_analysis` 之后，避免未评分就提前只保留一个候选。

## 默认运行方式

旧单链流程仍是默认路径：

```bash
bash run_loop.sh
```

启用树状搜索时显式打开配置：

```bash
ENABLE_TREE_SEARCH=true bash run_loop.sh
```

推荐起始参数：

```bash
MAX_SAMPLE_BRANCHES=4
MAX_SAMPLE_DEPTH=3
MAX_SAMPLE_BOUNDARIES=3
MAX_SAMPLE_CANDIDATES_TOTAL=10
MAX_NO_NEW_BOUNDARY_ROUNDS=2
ENABLE_BRANCH_BACKTRACK=true
ENABLE_ROOT_FORK=true
```

如果需要更接近旧单链行为，可关闭回溯与根节点 fork：

```bash
ENABLE_TREE_SEARCH=true ENABLE_BRANCH_BACKTRACK=false ENABLE_ROOT_FORK=false bash run_loop.sh
```

## 迁移兼容说明

旧输入 record 可以缺少树状搜索字段。`search_state_contract.py` 和阶段05状态更新会按兼容策略补齐：

1. 缺少 `search_root_id` 时，用样本 ID 或 prompt hash 生成稳定 root。
2. 缺少 `current_node_id` 时，旧未进化样本使用 root，已进化样本使用 round 与 prompt hash 生成节点。
3. 缺少预算字段时，使用 `MAX_SAMPLE_DEPTH` 和 `MAX_SAMPLE_CANDIDATES_TOTAL` 的默认值。
4. 缺少 `discovered_boundaries`、`recommended_next_axes`、`already_explored_axes` 时，按空列表处理。
5. `ENABLE_TREE_SEARCH=false` 时，旧 `update_records()` 契约和旧单链停止状态保持兼容。

新增 artifacts 不替代主链输出：

1. `state_updated.jsonl` 仍是下一轮主状态输入。
2. `search_graph.jsonl` 只用于复盘节点、边界轴、父子关系和选中状态。
3. `active_frontier.jsonl` 只在启用树状搜索时作为下一轮 `prepare_frontier_records.py` 的待扩展节点池。该步骤会把 `source_prompt` 覆盖为本轮顶层 `prompt`，并保留 `_frontier_context`，因此画像、分流、路由和进化共享同一批 frontier-expanded records。

## 主流程闭环状态与验收边界

当前 `run_loop.sh` 的树状搜索闭环已按 12 步接入：

1. 复制上一轮 `scored/state` 输入。
2. 启用树状搜索且存在上一轮 `active_frontier.jsonl` 时，生成 `frontier_records.jsonl`。
3. `profile_samples.py`、`select_evolution_candidates.py`、`operator_router.py` 和 `question_evolution.py` 均消费本轮同一批输入记录。
4. `validate_evolved_question.py` 后不再立即终选；所有候选先进入 `collect_answers.py`、`gen_rubric.py`、`scoring.py`。
5. `analyze_evolution_effect.py` 对候选全集输出 `is_new_boundary_for_sample`、`duplicate_boundary_for_sample` 和边界轴审计字段。
6. `candidate_selection.py` 在带 `effect_analysis` 的候选全集上做终选，保留新边界叶子并排除重复边界。
7. `update_sample_state.py` 消费终选后的带效果分析记录，更新 `search_graph.jsonl`、`active_frontier.jsonl`、分支预算和样本停止状态。

验收边界如下：

1. mock pipeline 验证结构闭环、字段传递、frontier 两轮回放、边界感知终选和状态更新，不代表真实 API 成本、模型输出质量或得分分布。
2. 真实实验验收仍需要在有 API、Bash 和目标模型服务的环境中执行小样本 dry run，检查实际 `experiments/YYYY-MM-DD/exp*/round_N/` 产物。
3. 当前全局停止计数以 `effect_analysis.is_new_boundary_for_sample == true` 且非重复边界为准，`MAX_GLOBAL_NEW_BOUNDARY_GAP` 控制连续无新增边界的全局停止。

## 验收覆盖

已建立的验证层包括：

1. schema/contract 测试：旧单链 record 补齐、graph/frontier fixture 与 schema 校验。
2. 决策层测试：能力轴画像兜底、已探索能力轴跳过、root fork 路由和旧路由兼容。
3. 生成层测试：当前分支扩展、root fork、parent fork、frontier overlay 和候选数预算门控。
4. 边界层测试：新边界、重复边界、能力轴不一致审计、边界叶子保留。
5. 状态层测试：预算扣减、root/parent frontier 生成、样本停止、禁用 fork 回退。
6. run loop 静态集成测试：12 步顺序、frontier 前置 overlay、后置边界感知终选、tree-search 参数、全局新增边界停止和 graph/frontier 输出。
7. tiny mock pipeline：无真实 API 的小样本闭环，验证旧单链产物、启用树状搜索后的 graph/frontier 产物、同组新边界/重复边界终选，以及第二轮 frontier 驱动画像和路由。
8. Bash 语法检查：新增 `.github/workflows/bash-syntax.yml`，在 Bash-compatible CI 中执行 `bash -n run_loop.sh`。

## 验收结论

阶段06的结构闭环验收目标已经满足：

1. 核心结构逻辑有 focused pytest 覆盖。
2. 禁用树状搜索时旧 mock pipeline 可运行。
3. 启用树状搜索时 tiny mock pipeline 可写出 `search_graph.jsonl` 和 `active_frontier.jsonl`，并用该 frontier 生成下一轮画像/路由输入。
4. `search_graph.jsonl` 可复盘 `sample_id`、root/node/parent、边界轴、命中状态、边界叶子和去重签名。
5. `active_frontier.jsonl` 可复盘下一步 root/parent fork、目标能力轴、剩余预算和来源停止状态。
6. 真实实验是否达到预期效果仍以带 API 的小样本 dry run 和后续实验指标为准，不能仅由 mock pipeline 推断。

## 已知风险

1. 真实 LLM 输出仍可能产生新能力轴表述，需要根据实验继续补充关键词归一化和 prompt 约束。
2. tiny mock pipeline 验证的是结构闭环，不代表真实 API 成本或效果分布。
3. `ENABLE_TREE_SEARCH` 默认仍为 `false`，正式实验需要显式开启并控制样本数和预算。
4. 旧实验输出目录未迁移；新 artifacts 只保证后续新轮次可写出。
5. Bash 语法检查已纳入 CI；当前本地 PowerShell 环境若仍找不到 `bash`，本地验证记录应保留该环境限制。

## 后续调参建议

1. 先用 `MAX_SAMPLE_CANDIDATES_TOTAL=6`、`MAX_SAMPLE_DEPTH=2` 做小样本 smoke run。
2. 确认 graph/frontier 结构稳定后，再恢复推荐默认值。
3. 优先观察每个样本的 `discovered_boundaries` 数量、重复边界比例和 `frontier_empty_stop` 触发频率。
4. 如果重复边界过多，收紧阶段04的去重签名或减少 `MAX_SAMPLE_BRANCHES`。
5. 如果过早停止，检查 `recommended_next_axes`、`already_explored_axes` 和 `MAX_NO_NEW_BOUNDARY_ROUNDS`。
