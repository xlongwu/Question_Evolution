## 1. 计划目标

本计划的目标不是重写 Question Evolution 流水线，而是在当前已经完成的：

```text
profile -> select -> router -> evolve -> validate -> select candidate
-> collect answers -> rubric -> scoring -> effect -> state update
```

基础上，增加一个“样本级搜索层”，使系统具备：

1. 单样本多边界探索；
2. 命中一个边界后的回溯继续搜索；
3. 单样本预算控制；
4. 边界去重；
5. 搜索轨迹沉淀。

最终产物从“每个样本一条主链”升级为“每个样本一个浅层搜索过程 + 多个边界叶子”。

---

## 2. 实施原则

### 2.1 总原则

1. 先增量、后增强，不做一步到位的大改。
2. 先让“根节点重新分叉”可用，再做“父节点回溯开兄弟分支”。
3. 先复用现有状态和脚本接口，再新增辅助文件。
4. 先保证现有 pipeline 能继续跑，再逐步增加搜索调度能力。

### 2.2 明确不做的事情

本计划不包含：

1. 重写 `collect_answers.py`；
2. 重写 `gen_rubric.py`；
3. 重写 `scoring.py`；
4. judge 可信度改造；
5. rubric drift 治理；
6. 一轮内一个样本多个分支同时进入评分闭环。

### 2.3 真正的最小可行增量

第一版只要求做到：

1. 一个样本命中边界后不直接整体停掉；
2. 样本可以从原始题重新开一个新能力轴分支；
3. 已命中过的边界不会重复计数；
4. 搜索行为可在状态和轨迹中复盘。

## 3. 当前基线约束

在正式改代码前，必须统一以下基线认知。

### 3.1 当前系统已有的能力

当前项目已经具备：

1. `profile_samples.py`：样本画像与 overscore 诊断；
2. `select_evolution_candidates.py`：样本分流；
3. `operator_router.py`：operator 路由；
4. `question_evolution.py`：单轮多候选生成；
5. `validate_evolved_question.py`：复杂度校验；
6. `candidate_selection.py`：候选选优；
7. `analyze_evolution_effect.py`：效果分析；
8. `update_sample_state.py`：状态更新；
9. `run_loop.sh`：按 round 执行完整闭环。

### 3.2 当前系统的真实短板

当前缺的是跨轮搜索能力，而不是单轮候选能力：

1. 多 candidate 无法跨轮保留为分支；
2. 一个边界命中后通常按样本整体停止；
3. 没有样本级 `discovered_boundaries`；
4. 没有显式 `frontier`；
5. 没有样本级预算；
6. 没有搜索轨迹文件；
7. 没有“根分叉”和“父节点分叉”的可执行状态机。

### 3.3 当前不能直接做的事情

短期内不能直接把：

1. `active_frontier.jsonl` 作为主输入；
2. `search_graph.jsonl` 作为唯一真相源；
3. `candidate_selection.py` 作为边界判定器；
4. 一轮多分支并行评分；
5. 完全取消 round 机制。

这些都要在后续阶段逐步过渡。

---

## 4. 总体实施路线

整个实施建议分为五个阶段：

1. 阶段0：字段契约与状态机扩展
2. 阶段1：根节点回溯分叉 MVP
3. 阶段2：搜索轨迹与 frontier 落盘
4. 阶段3：父节点回溯开兄弟分支
5. 阶段4：frontier 优先调度与实验验收

推荐执行方式：

```text
阶段0 -> 阶段1 -> 阶段2
```

完成这三步后即可形成第一版可用系统。

```text
阶段3 -> 阶段4
```

作为第二批增强能力，不建议提前混入第一轮开发。

## 6. 阶段1：根节点回溯分叉 MVP

### 6.1 阶段目标

实现第一版真正可用的树搜索能力：

1. 命中一个边界后不直接整体停止；
2. 若样本仍有未探索能力轴，则从根节点重新 fork；
3. 一个样本可以逐轮探索多个不同能力轴；
4. 已发现边界进入 `discovered_boundaries`。

这是整个项目最关键的第一阶段。

### 6.2 核心行为定义

第一版仅支持两种搜索路径：

1. `expand_current_branch`
2. `fork_from_root`

不支持：

1. `fork_from_parent`
2. 一轮多 frontier 并行评分

### 6.3 需要修改的文件

1. `profile_samples.py`
2. `operator_router.py`
3. `question_evolution.py`
4. `analyze_evolution_effect.py`
5. `update_sample_state.py`
6. `tests/test_stage03_operator_routing.py`
7. `tests/test_stage05_effect_state.py`
8. `tests/test_stage06_run_loop_integration.py`

### 6.4 详细实现任务

#### 6.4.1 `profile_samples.py`

新增输出字段：

1. `boundary_axis_candidates`
2. `next_best_axes`
3. `already_explored_axes`

实现要求：

1. `already_explored_axes` 从 `evolution_state.explored_axes` 读取；
2. `next_best_axes` 只输出未探索或可再次细分的轴；
3. 不推荐 operator，只推荐能力轴。

#### 6.4.2 `operator_router.py`

新增输入依赖：

1. `evolution_state.search_action`
2. `sample_profile.next_best_axes`
3. `evolution_state.discovered_boundaries`

新增输出字段：

1. `search_action`
2. `source_node_id`
3. `target_branch_id`
4. `target_boundary_axis`

实现要求：

1. 若 `search_action=fork_from_root`，`source_node_id` 必须指向 root；
2. 若已有 `boundary_axis` 命中，router 优先切换到其他轴；
3. `primary_operator` 应与 `target_boundary_axis` 一致。

#### 6.4.3 `question_evolution.py`

新增能力：

1. 根据 `source_node_id` 选择源题；
2. 当 `search_action=fork_from_root` 时，从 root prompt 开新题；
3. 新增 metadata 中的节点与分支信息；
4. `candidate_generation` 中记录 `source_node_id`、`branch_id`、`search_action`。

实现要求：

1. 第一阶段 root prompt 统一从 `meta_info.prompt_old` 或种子题读取；
2. 当前轮若为 root fork，不得沿当前 prompt 继续换壳；
3. 若缺 root prompt，直接降级为 `expand_current_branch` 并打 warning。

#### 6.4.4 `analyze_evolution_effect.py`

新增逻辑：

1. 从 metadata 读取 `boundary_axis`；
2. 结合 `expected_evaluation_focus` 与 candidate answer 轻量判断是否真的命中该轴；
3. 生成 `dedup_signature`；
4. 判断 `is_new_boundary_for_sample`。

去重规则第一版：

1. 同 `boundary_axis`
2. 同 `target_subclaim`
3. 同 `expected_qwen_failure`

满足以上三项则视为重复。

#### 6.4.5 `update_sample_state.py`

新增逻辑：

1. 更新 `explored_axes`；
2. 命中新边界时更新 `discovered_boundaries`；
3. 样本未达到边界上限时决定是否 `fork_from_root`；
4. 更新 `sample_budget_remaining`；
5. 当所有高价值轴探索完后停止样本。

核心状态机：

1. `effective_boundary_probe + 新边界 + 仍有未探索轴`
   -> `search_action=fork_from_root`
   -> `stop_status=continue_search`

2. `effective_boundary_probe + 新边界 + 无未探索轴`
   -> `search_action=stop_sample`
   -> `stop_status=all_axes_explored_stop`

3. `full_score_no_drop`
   -> `search_action=fork_from_root` 或 `continue_with_new_operator`
   视是否存在更优未探索轴而定

4. `invalid_complexity`
   -> 当前分支停止，但样本不一定停止

### 6.5 本阶段测试

必须新增的测试场景：

1. 命中首个边界后 root fork 成功；
2. 命中重复边界后不重复计入；
3. 已探索轴不再被优先路由；
4. 样本预算耗尽后停止；
5. root prompt 缺失时安全降级；
6. 现有 `local_tree_search_needed` 能平滑迁移到新状态机。

### 6.6 本阶段完成标准

完成标准：

1. 一个样本在多轮中可探索至少两个不同 `boundary_axis`；
2. `state_updated.jsonl` 中可看到 `discovered_boundaries`；
3. `effect_analysis.jsonl` 中可看到 `is_new_boundary_for_sample`；
4. tiny pipeline 冒烟测试通过；
5. 不修改 `collect_answers.py`、`gen_rubric.py`、`scoring.py`。

2. `experiments/.../round_N/active_frontier.jsonl`

2. `run_loop.sh`
3. `README.md`
4. `tests/test_stage06_tiny_pipeline_artifacts.py`

### 7.4 详细实现任务

新增：

1. 写 `search_graph.jsonl`
2. 写 `active_frontier.jsonl`

写入规则：

1. `search_graph.jsonl` 一条评分完成的样本记录对应一个 node 明细；

### 7.5 本阶段测试

新增测试：

1. 运行 tiny pipeline 后生成 `search_graph.jsonl`；
2. `active_frontier.jsonl` 字段完整；
3. 关闭写出开关时主流程不报错；
4. 轨迹文件与 `state_updated.jsonl` 中的摘要字段一致。

### 7.6 本阶段完成标准

完成标准：

## 8. 阶段3：父节点回溯开兄弟分支

### 8.1 阶段目标

在 root fork 稳定后，增加更细粒度的回溯能力：

1. 不只回根节点；
2. 可以从中间节点开兄弟分支；
3. 实现真正的浅层树结构。

### 8.2 前置条件

只有满足以下条件后才进入本阶段：

1. 阶段1、阶段2稳定；
2. `search_graph.jsonl` 已经稳定落盘；
3. 能根据 `node_id` 找回对应 prompt。

### 8.3 需要修改的文件

1. `question_evolution.py`
2. `operator_router.py`
3. `update_sample_state.py`
4. `validate_evolved_question.py`
5. `tests/test_stage03_operator_routing.py`
6. `tests/test_stage06_run_loop_integration.py`

### 8.4 详细实现任务

#### 8.4.1 节点 prompt 取回

必须先实现：

1. 通过 `node_id` 从 `search_graph.jsonl` 找回 source prompt；
2. 通过 `parent_node_id` 追溯父节点。

#### 8.4.2 `operator_router.py`

新增策略：

1. 若某个中间节点附近仍有未探索细层能力轴，则允许 `fork_from_parent`；
2. 若 root 上还有更粗粒度未探索轴，优先 root fork。

#### 8.4.3 `question_evolution.py`

新增：

1. `source_node_id` 解析为历史节点 prompt；
2. `parent_node_id` 正确写入 metadata；
3. 生成 `generated_node_id`。

### 8.5 本阶段测试

必须覆盖：

1. 从父节点成功分叉；
2. 根分叉和父分叉行为可区分；
3. source prompt 恢复失败时降级策略；
4. 兄弟分支不会污染已有边界统计。

### 8.6 本阶段完成标准

完成标准：

1. 样本可形成深度 2-3 的浅层搜索树；
2. 节点间父子关系可追踪；
3. root fork 与 parent fork 都可稳定运行。

## 9. 阶段4：frontier 优先调度与实验验收

### 9.1 阶段目标

在搜索结构稳定后，逐步减弱“每轮统一推进”的色彩，转向“frontier 优先”的样本调度方式。

### 9.2 本阶段不追求的事情

本阶段仍不做：

1. 全量无 round 调度器；
2. 分支并行多路评分；
3. 多进程图搜索引擎。

### 9.3 需要修改的文件

1. `run_loop.sh`
2. `select_evolution_candidates.py`
3. `operator_router.py`
4. `update_sample_state.py`
5. `README.md`
6. `tests/test_stage06_run_loop_integration.py`

### 9.4 详细实现任务

1. `run_loop.sh` 下一轮可优先基于 `active_frontier.jsonl` 选择样本的当前 source node；
2. `select_evolution_candidates.py` 对“仍在搜索中的样本”和“新进入搜索的样本”做不同分流；
3. `update_sample_state.py` 增加样本级 `no_new_boundary_rounds` 停止逻辑；
4. 统计新增指标：
   - `new_boundary_count`
   - `duplicate_boundary_count`
   - `sample_completed_count`

### 9.5 本阶段测试

1. frontier 文件参与下一轮决策；
2. 无新增边界若连续达到阈值则停样本；
3. 整个实验可输出边界数量和重复率汇总。

### 9.6 本阶段完成标准

1. round 仍保留，但 frontier 已成为真实调度依据之一；
2. 搜索停止更依赖样本级预算与边界增量，而不是单纯全局轮数；
3. 可输出按样本统计的多边界成果。

3. 去重签名生成；
4. `discovered_boundaries` 更新；
5. 预算递减；
6. 搜索停止条件。

### 11.2 集成测试层

至少补三类集成测试：

1. 单样本两轴探索闭环；
2. 命中重复边界后不重复沉淀；
3. 根分叉 + 无效分支 + 新边界沉淀组合路径。

### 11.3 tiny sample 冒烟测试

必须检查以下产物：

1. `effect_analysis.jsonl`
2. `state_updated.jsonl`
3. `search_graph.jsonl`

### 11.4 人工抽检点

每个阶段都要保留人工抽检：

1. 选 2-3 个样本看 `boundary_axis` 是否真的切换；
2. 检查 root fork 是否确实回到了原始题而非沿当前 prompt 换壳；
3. 检查重复边界是否被错误记成新边界；
4. 检查样本是否因状态机错误过早停止。

## 12. 风险与规避策略

### 12.1 风险一：状态过度复杂，导致主流程失稳

规避：

1. 第一阶段不做 parent fork；
2. 先只扩展字段，不改主流程输入语义；
3. 所有新字段都允许缺省。

### 12.2 风险二：边界去重误判

规避：

1. 第一版只做 strict dedup；
2. `is_new_boundary_for_sample=false` 时保留明细记录但不计成果；
3. 保留人工复核样本。

### 12.3 风险三：root fork 仍然沿旧路径换壳

规避：

1. `question_evolution.py` 明确使用 root prompt；
2. `validate_evolved_question.py` 增加“回溯生成一致性”校验；
3. 抽样对比 source prompt 与当前 prompt。

### 12.4 风险四：搜索预算失控

规避：

1. 新增 `MAX_SAMPLE_BOUNDARIES`；
2. 新增 `MAX_SAMPLE_CANDIDATES_TOTAL`；
3. 新增 `MAX_BRANCH_DEPTH`；
4. 新增 `MAX_NO_NEW_BOUNDARY_ROUNDS`。

## 15. 最终验收口径

最终是否算实施成功，以以下标准判断：

### 15.1 必须满足

1. 一个样本命中一个边界后不会必然停样本；
2. 同一样本至少可稳定探索两个不同能力轴；
3. 已发现边界可在 `state_updated.jsonl` 中结构化查看；
4. 搜索轨迹可在 `search_graph.jsonl` 中复盘；
5. 重复边界不会被重复记入有效成果。

### 15.2 不得出现

1. 题目重新大幅膨胀；
2. root fork 实际仍然沿当前题换壳；
3. 旧 pipeline 无法继续运行；
4. 现有 schema 和测试全面失效；
5. memory bank 被重复边界污染。

### 15.3 建议观察指标

1. 单样本平均有效边界数；
2. 新边界率；
3. 重复边界率；
4. 每个样本平均消耗候选数；
5. 每轮 `new_boundary_count`；
6. 有效边界按 `boundary_axis` 的分布。

## 16. 结论

这份实施计划的核心是：

1. 第一版只做真正必要且完全可落地的能力；
2. 所有关键判断尽量放在评分后做，而不是在候选选择阶段提前猜；
3. 所有新能力围绕当前 `evolution_state + metadata + effect_analysis` 增量扩展；
4. 先把 root fork 和边界去重做稳，再进入更复杂的 parent fork 和 frontier 调度。

按本计划推进，可以在不推翻现有 V3 项目的前提下，逐步把系统升级为“链式主流程 + 样本级浅层树搜索 + 回溯式多边界探索”。
